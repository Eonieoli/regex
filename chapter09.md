## 9.1 전후방 탐색이란 무엇인가

> 💡 **한 줄 요약**: 전후방 탐색은 "주변을 살펴보기만 하고, 실제로 문자를 소비하지 않는" 특수한 매칭 기법입니다.

### 직관적 이해

여러분이 도서관에서 책을 찾고 있다고 상상해봅시다. 사서에게 이렇게 요청합니다:

> "**'파이썬'이라고 적힌 책 옆에 꽂혀 있는 '정규표현식' 책**을 찾아주세요."

이 요청에서 핵심은 무엇일까요? 여러분이 실제로 빌리고 싶은 책은 **'정규표현식' 책**입니다. '파이썬' 책은 단지 *위치를 확인하기 위한 조건*일 뿐, 빌리려는 대상이 아닙니다. '파이썬' 책은 확인만 하고 제자리에 둡니다.

전후방 탐색(lookaround)이 바로 이런 동작을 합니다. **주변 문맥을 조건으로 확인하되, 확인한 부분은 매칭 결과에 포함시키지 않습니다.** 말 그대로 "둘러보기(look around)"만 하는 것입니다.

왜 이런 기능이 필요할까요? 지금까지 배운 일반적인 패턴은 매칭한 모든 문자를 "소비"합니다. 즉, 매칭된 문자는 결과에 포함되고, 엔진은 그 문자를 지나서 다음 위치로 이동합니다. 하지만 실무에서는 이런 요구사항이 빈번합니다:

- "가격 앞에 붙은 `$` 기호 없이 **숫자만** 추출하고 싶다"
- "`.txt`로 끝나는 파일명에서 **확장자를 제외한 이름만** 가져오고 싶다"
- "비밀번호가 대문자, 소문자, 숫자를 **모두 포함하는지** 한 번에 검증하고 싶다"

이 모든 경우에 전후방 탐색이 핵심 도구가 됩니다.

### 핵심 개념 설명

> 📝 **제로 너비 단언(Zero-width Assertion)**: 문자열에서 특정 **위치(position)**가 조건을 만족하는지 확인하되, 실제 문자를 소비하지 않는 패턴 요소. 매칭 결과의 길이에 기여하지 않으므로 "너비가 0(zero-width)"이라고 합니다.

사실 여러분은 이미 제로 너비 단언을 사용해본 적이 있습니다.

> 🔗 **연결**: Chapter 2에서 배운 앵커(`^`, `$`, `\A`, `\Z`)와 Chapter 3에서 배운 단어 경계(`\b`)가 바로 제로 너비 단언입니다. 이들은 문자 자체가 아니라 *위치*를 매칭합니다.

전후방 탐색은 이 제로 너비 단언의 개념을 **범용적으로 확장**한 것입니다. `^`는 "문자열의 시작 위치"만 확인할 수 있고, `\b`는 "단어 경계 위치"만 확인할 수 있지만, 전후방 탐색은 **임의의 패턴**을 조건으로 사용하여 위치를 확인할 수 있습니다.

전후방 탐색은 크게 네 가지 종류가 있습니다:

| 종류 | 문법 | 의미 | 방향 |
|------|------|------|------|
| 긍정 전방 탐색 | `(?=...)` | 앞(오른쪽)에 `...`이 **있는** 위치 | → |
| 부정 전방 탐색 | `(?!...)` | 앞(오른쪽)에 `...`이 **없는** 위치 | → |
| 긍정 후방 탐색 | `(?<=...)` | 뒤(왼쪽)에 `...`이 **있는** 위치 | ← |
| 부정 후방 탐색 | `(?<!...)` | 뒤(왼쪽)에 `...`이 **없는** 위치 | ← |

여기서 "앞"과 "뒤"의 방향이 헷갈릴 수 있습니다. 정규표현식 엔진은 문자열을 **왼쪽에서 오른쪽으로** 읽어나갑니다. 따라서:
- **전방(ahead)** = 현재 위치에서 **오른쪽** = 아직 읽지 않은 쪽 = "앞으로 나아갈 방향"
- **후방(behind)** = 현재 위치에서 **왼쪽** = 이미 읽은 쪽 = "지나온 방향"

**"소비하지 않는다(zero-width)"는 것이 정확히 무슨 의미일까요?**

일반적인 패턴 매칭에서 엔진이 `abc`를 매칭하면, 엔진의 "커서"는 `a`, `b`, `c`를 차례로 지나가며 3칸 앞으로 전진합니다. 이 세 문자는 "소비"되었고, 매칭 결과에 포함됩니다.

반면 전후방 탐색에서는 엔진이 조건 패턴을 확인한 뒤, **커서를 원래 위치로 되돌려놓습니다.** 마치 고개를 돌려 옆을 힐끗 본 다음 다시 정면을 바라보는 것과 같습니다. 확인은 했지만, 이동하지 않았으므로 아무것도 소비되지 않습니다.

```
일반 매칭:   "price: $100"에서 \$\d+ 매칭
             커서 이동: $ → 1 → 0 → 0
             결과: "$100" ($ 포함)

전방 탐색:   "price: $100"에서 (?=\$)\d+ 는 의도대로 동작하지 않음
             (전방 탐색 자체를 올바르게 쓰려면 다른 방식이 필요 — 이어서 배웁니다)
```

이제 각 종류를 하나씩 살펴보겠습니다.

### Section 요약

- 전후방 탐색(lookaround)은 주변 문맥을 확인하되 문자를 소비하지 않는 제로 너비 단언이다
- 이미 배운 `^`, `$`, `\b`도 제로 너비 단언의 일종이며, 전후방 탐색은 이를 범용적으로 확장한 것이다
- 전방 탐색은 오른쪽(앞으로 진행할 방향)을, 후방 탐색은 왼쪽(이미 지나온 방향)을 확인한다
- 네 가지 종류가 있다: 긍정/부정 × 전방/후방

## 9.2 긍정 전방 탐색: `(?=...)`

> 💡 **한 줄 요약**: `(?=...)`는 "바로 오른쪽에 `...`이 있는 위치"를 매칭하되, `...` 부분은 결과에 포함시키지 않습니다.

### 직관적 이해

카페에서 메뉴판을 보고 있다고 해봅시다. "아이스가 붙은 음료의 **기본 이름만** 알고 싶다"고 합니다:

- "아이스 아메리카노" → **아메리카노**
- "아이스 라떼" → **라떼**
- "핫 초코" → (제외)

여기서 "아이스"는 조건이지, 추출 대상이 아닙니다. 긍정 전방 탐색은 이런 패턴을 표현합니다: **"뒤에 특정 패턴이 따라오는 것만 골라내되, 그 패턴 자체는 결과에서 빼라."**

물론 위 예시에서는 "아이스" 뒤에 이름이 오니 후방 탐색이 더 적절하겠지만 — 개념을 이해하기 위해 전방 탐색부터 살펴보겠습니다.

### 핵심 개념 설명

긍정 전방 탐색의 문법은 `(?=패턴)`입니다.

> 📝 **긍정 전방 탐색(Positive Lookahead)** `(?=...)`: 현재 위치에서 오른쪽 방향으로 `...` 패턴이 매칭되는지 확인합니다. 조건이 참이면 현재 위치에서의 매칭이 성공하고, 커서는 이동하지 않습니다.

> 🔗 **연결**: Chapter 8에서 배운 비캡처 그룹 `(?:...)`의 문법과 비교해보세요. 둘 다 `(?`로 시작하지만, 비캡처 그룹은 `(?:...)`이고 전방 탐색은 `(?=...)`입니다. 비캡처 그룹은 문자를 소비하고, 전방 탐색은 소비하지 않습니다.

가장 간단한 예시로 시작하겠습니다:

```python
import re

# 전방 탐색 없이: "원"이 포함된 전체를 매칭
result = re.findall(r'\d+원', '사과 3000원 바나나 2000원 포도 5000원')
print(result)  # ['3000원', '2000원', '5000원']

# 전방 탐색 사용: "원" 앞의 숫자만 매칭 ("원"은 소비하지 않음)
result = re.findall(r'\d+(?=원)', '사과 3000원 바나나 2000원 포도 5000원')
print(result)  # ['3000', '2000', '5000']
```

두 번째 패턴 `\d+(?=원)`의 동작을 단계별로 살펴봅시다:

1. `\d+`가 연속된 숫자를 매칭합니다 (예: `3000`)
2. `(?=원)`이 현재 위치 바로 오른쪽에 `원`이 있는지 **확인만** 합니다
3. `원`이 있으면 → 매칭 성공! 하지만 `원`은 결과에 포함되지 않습니다
4. `원`이 없으면 → 매칭 실패, 다른 위치에서 재시도

### 상세 예시

> 🔍 **예시 9.2.1: 파일 확장자 앞의 파일명 추출**
>
> **상황**: 파일명 목록에서 `.py`로 끝나는 파일의 이름 부분(확장자 제외)만 추출하고 싶습니다.
>
> ```python
> import re
> 
> files = 'main.py config.json utils.py readme.md test.py'
> 
> # 전방 탐색으로 .py 앞의 파일명만 추출
> result = re.findall(r'\b\w+(?=\.py\b)', files)
> print(result)  # ['main', 'utils', 'test']
> ```
>
> **풀이/설명**:
> - `\b` — 단어 경계에서 시작 (파일명의 시작)
> - `\w+` — 하나 이상의 단어 문자를 매칭 (파일명 부분)
> - `(?=\.py\b)` — 바로 뒤에 `.py`와 단어 경계가 있는지 확인 (하지만 소비하지 않음)
>
> **핵심 포인트**: `.py`는 조건으로만 사용되었고, 매칭 결과에는 파일명(`main`, `utils`, `test`)만 포함됩니다. `\.py` 뒤에 `\b`를 넣어 `.pyc` 같은 다른 확장자에 잘못 매칭되는 것도 방지했습니다.

> 🔍 **예시 9.2.2: 쉼표 삽입 — 천 단위 구분 위치 찾기**
>
> **상황**: 큰 숫자에 천 단위 쉼표를 넣고 싶습니다. `1234567`을 `1,234,567`로 바꾸려면?
>
> ```python
> import re
> 
> number = '1234567890'
> 
> # 전방 탐색: "뒤에 3자리 숫자 그룹이 하나 이상 따라오고, 그 뒤가 끝인 위치"
> # 후방 탐색도 필요하지만, 우선 전방 탐색의 역할만 봅시다
> result = re.sub(r'(?=(\d{3})+$)', ',', number)
> print(result)  # ',1,234,567,890' — 아직 완벽하지 않습니다!
> ```
>
> 이 예시에서 `(?=(\d{3})+$)`는 "오른쪽에 3자리 숫자 묶음이 하나 이상 있고 그 다음이 문자열 끝인 위치"를 찾습니다. 하지만 맨 앞에도 쉼표가 들어가는 문제가 있습니다. 이 문제는 Section 9.6에서 후방 탐색과 조합하여 완벽하게 해결합니다.
>
> **핵심 포인트**: 전방 탐색은 *위치*를 매칭하므로, `re.sub()`과 함께 쓰면 **특정 위치에 문자를 삽입**하는 효과를 낼 수 있습니다. 전방 탐색 자체가 소비하는 문자가 없으므로, 치환 시 원래 문자는 그대로 유지되고 치환 문자열만 삽입됩니다.

> ⚠️ **주의**: 전방 탐색에서 가장 흔한 실수 — 전방 탐색의 위치를 잘못 배치하기
>
> ```python
> # 의도: "원" 뒤에 오는 숫자를 찾고 싶다
> # 잘못된 시도: 전방 탐색을 패턴 앞에 배치
> re.findall(r'(?=원)\d+', '가격 3000원 할인 500원')
> # 결과: [] — 아무것도 매칭되지 않음!
> ```
>
> **왜 실패할까요?** `(?=원)`은 "현재 위치 오른쪽에 `원`이 있어야 한다"는 뜻입니다. 하지만 바로 그 위치에서 `\d+`(숫자)도 시작해야 합니다. `원`이 있는 위치에서 숫자가 시작될 수는 없으므로 매칭이 실패합니다.
>
> **전방 탐색은 보통 패턴의 뒤쪽(오른쪽)에 배치합니다.** "먼저 무언가를 매칭한 다음, 그 뒤에 특정 조건이 있는지 확인"하는 구조가 자연스럽습니다.

### Section 요약

- `(?=...)`는 현재 위치의 오른쪽에 특정 패턴이 있는지 확인하되, 문자를 소비하지 않는다
- 주로 "뒤에 특정 패턴이 따라오는 부분만 매칭"하는 용도로 사용한다
- `re.sub()`과 결합하면 특정 위치에 문자를 삽입하는 효과를 낼 수 있다
- 전방 탐색은 보통 메인 패턴의 **뒤쪽**에 배치한다

## 9.3 부정 전방 탐색: `(?!...)`

> 💡 **한 줄 요약**: `(?!...)`는 "바로 오른쪽에 `...`이 **없는** 위치"를 매칭하며, 긍정 전방 탐색의 반대 논리입니다.

### 직관적 이해

이번에는 식당 알레르기 확인 상황을 떠올려봅시다. "**땅콩이 들어가지 않은** 메뉴를 모두 보여주세요"라고 요청하는 것과 같습니다. 여기서 "땅콩"은 **배제 조건**입니다 — 땅콩이 없는 것을 찾는 것이지, 땅콩을 찾는 것이 아닙니다.

부정 전방 탐색은 이렇게 **특정 패턴이 뒤따르지 않는 위치**를 찾습니다.

### 핵심 개념 설명

> 📝 **부정 전방 탐색(Negative Lookahead)** `(?!...)`: 현재 위치에서 오른쪽 방향으로 `...` 패턴이 매칭되**지 않는** 위치를 찾습니다. 조건이 거짓(패턴 불일치)이면 매칭이 성공합니다.

긍정 전방 탐색 `(?=...)`과 정반대의 논리입니다:

| 전방 탐색 종류 | 오른쪽에 패턴이 있을 때 | 오른쪽에 패턴이 없을 때 |
|------------|-----------------|-----------------|
| 긍정 `(?=...)` | ✅ 매칭 성공 | ❌ 매칭 실패 |
| 부정 `(?!...)` | ❌ 매칭 실패 | ✅ 매칭 성공 |

```python
import re

text = 'apple.py apple.txt banana.py cherry.md'

# 긍정: .py로 끝나는 파일명
re.findall(r'\b\w+(?=\.py\b)', text)
# ['apple', 'banana']

# 부정: .py로 끝나지 않는 파일명 — 단, 확장자가 있는 것만
re.findall(r'\b\w+(?!\.py\b)(?=\.\w+\b)', text)
# 의도는 맞지만 패턴이 복잡해집니다. 더 나은 방법을 예시에서 봅시다.
```

### 상세 예시

> 🔍 **예시 9.3.1: 특정 단어 뒤에 오지 않는 패턴 찾기**
>
> **상황**: 텍스트에서 `http`로 시작하되 `https`는 아닌(즉, 보안되지 않은) URL의 시작부를 찾고 싶습니다.
>
> ```python
> import re
> 
> text = 'http://example.com https://secure.com http://test.org'
> 
> # "http" 다음에 "s"가 오지 않는 것만 매칭
> result = re.findall(r'http(?!s)://\S+', text)
> print(result)  # ['http://example.com', 'http://test.org']
> ```
>
> **풀이/설명**:
> - `http` — 리터럴 "http"를 매칭
> - `(?!s)` — 바로 뒤에 "s"가 **없는지** 확인 (소비하지 않음)
> - `://\S+` — 이어서 "://"와 공백이 아닌 문자들을 매칭
>
> **핵심 포인트**: `(?!s)`는 `http` 바로 다음 위치에서 `s`가 오지 않는지만 확인합니다. `https`의 경우 `(?!s)`가 실패하므로 전체 매칭이 실패하고, `http://`의 경우 `(?!s)` 다음에 `:`가 오므로 성공합니다.

> 🔍 **예시 9.3.2: 특정 확장자를 제외한 파일명 찾기**
>
> **상황**: 파일 목록에서 `.bak`이나 `.tmp` 확장자가 **아닌** 파일만 찾고 싶습니다.
>
> ```python
> import re
> 
> files = 'data.csv report.bak image.png cache.tmp notes.txt'
> 
> # 파일명.확장자 형태에서, 확장자가 bak이나 tmp가 아닌 것
> result = re.findall(r'\b\w+\.(?!bak\b|tmp\b)\w+\b', files)
> print(result)  # ['data.csv', 'image.png', 'notes.txt']
> ```
>
> **풀이/설명**:
> - `\b\w+\.` — 단어 경계 + 파일명 + 마침표
> - `(?!bak\b|tmp\b)` — 마침표 다음에 `bak`이나 `tmp`(단어 경계까지)가 **오지 않는지** 확인
> - `\w+\b` — 실제 확장자를 매칭
>
> **핵심 포인트**: 부정 전방 탐색 안에서 `|`(OR)를 사용하여 여러 배제 조건을 지정할 수 있습니다. `\b`를 넣어 `bak`이 `backup`과 같은 더 긴 확장자에 잘못 매칭되는 것을 방지했습니다.

> ⚠️ **주의**: 부정 전방 탐색의 범위를 정확히 이해하기
>
> ```python
> # 의도: "test"라는 단어가 아닌 모든 4글자 단어를 찾기
> # 잘못된 시도:
> re.findall(r'(?!test)\b\w{4}\b', 'test best rest nest')
> # 결과: ['test', 'best', 'rest', 'nest'] — test도 포함됨!
> ```
>
> **왜 이런 일이 벌어질까요?** `(?!test)`는 **현재 위치에서** `test`가 시작되지 않는지를 확인합니다. 하지만 `\b`가 `(?!test)` 뒤에 있으므로, 엔진이 단어 시작이 아닌 위치에서도 `(?!test)`를 시도할 수 있습니다.
>
> **올바른 방법**:
> ```python
> re.findall(r'\b(?!test\b)\w{4}\b', 'test best rest nest')
> # 결과: ['best', 'rest', 'nest'] — 정확!
> ```
>
> `\b`를 먼저 써서 단어 경계에 위치를 고정한 뒤, `(?!test\b)`로 "이 단어 경계에서 시작하는 것이 `test`가 아닌지" 확인합니다.

### Section 요약

- `(?!...)`는 현재 위치의 오른쪽에 특정 패턴이 **없는지** 확인하는 부정 조건이다
- 특정 패턴을 **배제**하는 용도로 강력하다 (예: `https`가 아닌 `http`, 특정 확장자 제외)
- 부정 전방 탐색 안에서 `|`로 여러 배제 조건을 결합할 수 있다
- 부정 전방 탐색의 위치 배치에 주의해야 한다 — 앵커(`\b`)와의 순서가 중요하다

## 9.4 긍정 후방 탐색: `(?<=...)`

> 💡 **한 줄 요약**: `(?<=...)`는 "바로 왼쪽에 `...`이 있는 위치"를 매칭하며, 전방 탐색의 반대 방향입니다.

### 직관적 이해

이번에는 은행 거래 내역서를 살펴보는 상황입니다. "입금"이라고 표시된 항목의 **금액만** 알고 싶습니다:

```
입금: 50,000원
출금: 30,000원
입금: 120,000원
출금: 15,000원
```

여기서 "입금:"은 찾는 조건이지, 추출 대상이 아닙니다. "입금:" **뒤에 오는** 금액만 원합니다. 이때 후방 탐색이 필요합니다: **왼쪽(이미 지나온 방향)의 문맥을 조건으로 확인**합니다.

왜 "후방(behind)"이라고 할까요? 정규표현식 엔진은 왼쪽에서 오른쪽으로 진행하므로, 왼쪽은 이미 지나온 "뒤(behind)" 방향입니다.

### 핵심 개념 설명

> 📝 **긍정 후방 탐색(Positive Lookbehind)** `(?<=...)`: 현재 위치에서 왼쪽 방향으로 `...` 패턴이 이미 매칭되어 있는지 확인합니다. 조건이 참이면 매칭이 성공하고, 역시 문자를 소비하지 않습니다.

```python
import re

text = '입금: 50,000원\n출금: 30,000원\n입금: 120,000원'

# "입금: " 뒤에 오는 숫자+쉼표 조합만 추출
result = re.findall(r'(?<=입금: )[\d,]+', text)
print(result)  # ['50,000', '120,000']
```

**⚠️ 파이썬에서 후방 탐색의 중요한 제약 사항:**

파이썬의 `re` 모듈에서 후방 탐색(`(?<=...)`, `(?<!...)`) 안의 패턴은 **고정 길이(fixed-length)**여야 합니다. 즉, 매칭되는 문자의 수가 사전에 결정 가능해야 합니다.

```python
# ✅ 허용: 고정 길이 패턴
r'(?<=abc)\d+'       # "abc"는 항상 3글자 — OK
r'(?<=\d{3})\w+'     # 항상 3자리 숫자 — OK
r'(?<=ab|cd)\d+'     # "ab" 또는 "cd", 둘 다 2글자 — OK

# ❌ 불허: 가변 길이 패턴
r'(?<=\d+)\w+'       # \d+는 1글자일 수도, 10글자일 수도 — 에러!
r'(?<=a*b)\d+'       # a*는 0글자일 수도, 여러 글자일 수도 — 에러!
r'(?<=ab|cde)\d+'    # "ab"는 2글자, "cde"는 3글자 — 에러!
```

이 제약은 파이썬 `re` 모듈의 NFA 엔진이 후방 탐색을 처리하는 방식 때문입니다. 고정 길이여야 엔진이 "현재 위치에서 정확히 몇 글자 왼쪽을 확인하면 되는지"를 알 수 있습니다.

> 💬 **참고**: 이 고정 길이 제약은 `re` 모듈의 한계입니다. Chapter 17에서 다루는 서드파티 `regex` 모듈은 가변 길이 후방 탐색을 지원합니다.

### 상세 예시

> 🔍 **예시 9.4.1: 통화 기호 뒤의 금액 추출**
>
> **상황**: 다양한 통화가 섞인 텍스트에서 달러(`$`) 금액만 추출하고 싶습니다.
>
> ```python
> import re
> 
> text = 'Price: $49.99, Tax: $5.00, Total: €54.99, Tip: $10.00'
> 
> # $ 기호 바로 뒤의 숫자(소수점 포함)만 추출
> result = re.findall(r'(?<=\$)\d+(?:\.\d{2})?', text)
> print(result)  # ['49.99', '5.00', '10.00']
> ```
>
> **풀이/설명**:
> - `(?<=\$)` — 바로 왼쪽에 `$`가 있는지 확인 (소비하지 않음, `$`는 메타문자이므로 `\$`로 이스케이프)
> - `\d+` — 정수 부분 매칭
> - `(?:\.\d{2})?` — 소수점과 2자리가 선택적으로 따라옴 (비캡처 그룹 사용)
>
> **핵심 포인트**: `$` 기호가 결과에 포함되지 않으므로, 추출된 문자열을 바로 숫자로 변환(`float()`)할 수 있습니다. `€54.99`는 `$`가 아니므로 자연스럽게 제외됩니다.

> 🔍 **예시 9.4.2: 특정 HTML 태그 안의 내용 추출**
>
> **상황**: `<b>` 태그로 감싸진 텍스트의 내용만 추출하고 싶습니다.
>
> ```python
> import re
> 
> html = '<b>중요</b>한 내용과 <b>핵심</b> 포인트, 그리고 <i>참고</i> 사항'
> 
> # <b> 뒤, </b> 앞의 내용만 추출
> result = re.findall(r'(?<=<b>).+?(?=</b>)', html)
> print(result)  # ['중요', '핵심']
> ```
>
> **풀이/설명**:
> - `(?<=<b>)` — 왼쪽에 `<b>`가 있는지 확인 (고정 길이 3글자 — OK)
> - `.+?` — 최소한의 문자를 매칭 (비탐욕적)
> - `(?=</b>)` — 오른쪽에 `</b>`가 있는지 확인
>
> 🔗 **연결**: Chapter 4에서 배운 비탐욕적 수량자 `.+?`가 여기서 핵심입니다. 탐욕적 `.+`를 사용하면 첫 `<b>` 뒤에서 마지막 `</b>` 앞까지 전부를 매칭해버려, `중요</b>한 내용과 <b>핵심`이 되어버립니다.
>
> **핵심 포인트**: 후방 탐색과 전방 탐색을 **동시에** 사용하여 "왼쪽에 A가 있고, 오른쪽에 B가 있는" 위치의 내용을 정밀하게 추출할 수 있습니다. 이 조합은 매우 강력하며 자주 사용됩니다.

> ⚠️ **주의**: 후방 탐색의 고정 길이 제약을 잊지 마세요
>
> ```python
> # 가변 길이 태그를 후방 탐색에 사용하려고 하면:
> re.findall(r'(?<=<\w+>).+?(?=</\w+>)', '<b>text</b>')
> # re.error: look-behind requires fixed-width pattern
> 
> # 해결: 태그명을 구체적으로 지정
> re.findall(r'(?<=<b>).+?(?=</b>)', '<b>text</b>')  # OK
> ```
>
> `<\w+>`는 태그명 길이가 가변적이므로 후방 탐색에 사용할 수 없습니다. 구체적인 태그명을 알고 있다면 직접 적는 것이 가장 확실합니다.

> 🔬 **Deep Dive**: 후방 탐색의 내부 동작
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 후방 탐색은 전방 탐색과 동작 방식이 근본적으로 다릅니다. 전방 탐색은 현재 위치에서 "앞(오른쪽)을 내다보면" 되므로 엔진의 자연스러운 진행 방향과 같습니다. 하지만 후방 탐색은 **역방향**으로 확인해야 합니다.
>
> 파이썬 `re` 모듈은 이를 처리하기 위해 다음과 같은 전략을 사용합니다:
>
> 1. 후방 탐색 패턴의 **고정 길이 `n`**을 계산합니다
> 2. 현재 위치에서 **`n`칸 왼쪽으로 이동**합니다
> 3. 그 위치에서 패턴을 정방향으로 매칭해봅니다
> 4. 매칭이 성공하면 → 후방 탐색 조건 충족
> 5. 커서를 원래 위치로 복원합니다
>
> 이것이 바로 "고정 길이"가 요구되는 이유입니다. 패턴 길이가 가변적이면 몇 칸 왼쪽으로 이동해야 하는지 결정할 수 없기 때문입니다. 이 내부 동작은 Chapter 15에서 엔진 원리를 학습할 때 더 깊이 다룹니다.

### Section 요약

- `(?<=...)`는 현재 위치의 왼쪽에 특정 패턴이 있는지 확인하는 긍정 조건이다
- 파이썬 `re` 모듈에서 후방 탐색의 패턴은 **고정 길이**여야 한다 (`+`, `*`, `?` 등 가변 수량자 사용 불가)
- 후방 탐색과 전방 탐색을 결합하면 "양쪽 문맥 조건"을 지정하여 매우 정밀한 추출이 가능하다
- `|`(OR)를 후방 탐색 안에서 사용할 때, 각 선택지의 길이가 동일해야 한다

## 9.5 부정 후방 탐색: `(?<!...)`

> 💡 **한 줄 요약**: `(?<!...)`는 "바로 왼쪽에 `...`이 **없는** 위치"를 매칭하며, 긍정 후방 탐색의 반대 논리입니다.

### 직관적 이해

이번에는 교정(proofreading) 상황입니다. 문장에서 마침표(`.`) 뒤에 공백이 오는 것은 정상이지만, **약어가 아닌 곳에서** 마침표 뒤에 공백 없이 바로 대문자가 오는 것은 오류입니다. "Mr."이나 "Dr." 같은 약어 뒤에서는 이런 패턴이 정상이니 무시해야 합니다.

부정 후방 탐색은 이렇게 **왼쪽에 특정 패턴이 없는 위치**를 찾는 데 사용합니다.

### 핵심 개념 설명

> 📝 **부정 후방 탐색(Negative Lookbehind)** `(?<!...)`: 현재 위치에서 왼쪽 방향으로 `...` 패턴이 매칭되**지 않는** 위치를 찾습니다.

네 가지 전후방 탐색의 관계를 정리하면:

| | 긍정 (있으면 성공) | 부정 (없으면 성공) |
|---|---|---|
| **전방 (오른쪽 확인)** | `(?=...)` | `(?!...)` |
| **후방 (왼쪽 확인)** | `(?<=...)` | `(?<!...)` |

```python
import re

# 긍정 후방: "$" 뒤의 숫자 (달러 가격)
re.findall(r'(?<=\$)\d+', 'Items: $50, €30, $100')
# ['50', '100']

# 부정 후방: "$"가 아닌 것 뒤의 숫자 (달러가 아닌 가격)
re.findall(r'(?<!\$)\d+', 'Items: $50, €30, $100')
# ['0', '30', '00']  — 주의! 예상과 다를 수 있음
```

위 부정 후방 탐색의 결과가 예상과 다를 수 있습니다. `$50`에서 `5` 앞에는 `$`가 있으므로 `5`는 제외되지만, `0` 앞에는 `5`가 있으므로(달러가 아님) `0`은 매칭됩니다. 이것이 부정 후방 탐색에서 자주 발생하는 함정입니다.

```python
# 더 정확한 패턴: 달러 기호가 아닌 통화 뒤의 전체 숫자
re.findall(r'(?<!\$)\b\d+', 'Items: $50, €30, $100')
# ['30']  — \b(단어 경계)를 추가하여 숫자의 시작부터 매칭
```

> ⚠️ **주의**: 부정 후방 탐색도 긍정 후방 탐색과 동일한 **고정 길이 제약**이 적용됩니다.

### 상세 예시

> 🔍 **예시 9.5.1: 이스케이프되지 않은 특수문자 찾기**
>
> **상황**: 설정 파일에서 `\`로 이스케이프되지 않은 `#` 문자만 찾고 싶습니다 (이스케이프된 `\#`는 리터럴이므로 무시).
>
> ```python
> import re
> 
> text = r'color = blue # 주석 입니다 \# 이것은 리터럴 해시 # 또 다른 주석'
> 
> # 백슬래시가 앞에 없는 # 찾기
> result = re.findall(r'(?<!\\)#', text)
> print(result)  # ['#', '#']  — 이스케이프 안 된 # 2개만 찾음
> ```
>
> **풀이/설명**:
> - `(?<!\\)` — 바로 왼쪽에 `\`가 **없는지** 확인. raw string에서 `\\`은 리터럴 `\` 한 개를 의미합니다
> - `#` — 해시 문자 매칭
>
> **핵심 포인트**: 이스케이프 문자 처리는 프로그래밍에서 매우 흔한 작업입니다. 부정 후방 탐색은 "앞에 이스케이프 문자가 없는" 특수문자를 찾는 데 이상적입니다.

> 🔍 **예시 9.5.2: 특정 접두사가 없는 단어 찾기**
>
> **상황**: 로그에서 "Error"라는 단어를 찾되, "NoError"나 "NonError" 같은 합성어는 제외하고 싶습니다.
>
> ```python
> import re
> 
> log = 'Error occurred. NoError state. Error found. NonError case.'
> 
> # "Error" 앞에 영문자가 없는 것만 찾기 (= 독립된 Error)
> result = re.findall(r'(?<![a-zA-Z])Error', log)
> print(result)  # ['Error', 'Error']
> ```
>
> **풀이/설명**:
> - `(?<![a-zA-Z])` — 바로 왼쪽에 영문자가 **없는지** 확인
> - `Error` — 리터럴 "Error" 매칭
>
> `NoError`의 `Error` 앞에는 `o`(영문자)가 있으므로 후방 탐색이 실패하여 제외됩니다. 독립된 `Error` 앞에는 공백이나 문자열 시작이 있으므로 성공합니다.
>
> 💬 **참고**: 이 경우 `\bError`로도 비슷한 효과를 얻을 수 있습니다. 하지만 부정 후방 탐색은 단어 경계보다 더 세밀한 조건을 지정할 수 있어 유연합니다. 예를 들어 "숫자 뒤가 아닌 Error만"이나 "특정 기호 뒤가 아닌 Error만" 같은 조건은 `\b`만으로는 표현할 수 없습니다.

### Section 요약

- `(?<!...)`는 현재 위치의 왼쪽에 특정 패턴이 **없는지** 확인하는 부정 조건이다
- 이스케이프되지 않은 특수문자 찾기, 특정 접두사 없는 단어 찾기 등에 유용하다
- 긍정 후방 탐색과 마찬가지로 **고정 길이 제약**이 적용된다
- 부정 후방 탐색 사용 시 숫자의 중간 부분이 매칭되는 등 예상외의 결과에 주의해야 한다

## 9.6 전후방 탐색 조합 패턴

> 💡 **한 줄 요약**: 전방과 후방 탐색을 조합하면 복잡한 문맥 조건을 하나의 패턴으로 표현할 수 있으며, 특히 비밀번호 검증과 같은 "복합 조건 AND" 패턴에 강력합니다.

### 직관적 이해

지금까지 전후방 탐색을 하나씩 배웠습니다. 이제 이것들을 **조합**하여 진정으로 강력한 패턴을 만들어봅시다.

실생활에서 "이 음식은 글루텐 프리이면서(AND) 비건이면서(AND) 저칼로리인 것"처럼 여러 조건을 동시에 만족하는 것을 찾는 경우가 많습니다. 일반적인 정규표현식에서는 AND 논리를 직접 표현하기 어렵지만, 전방 탐색을 여러 개 연결하면 바로 이 AND 조건을 구현할 수 있습니다.

### 핵심 개념 설명

**조합 패턴 1: 전방 + 후방을 동시에 사용하여 양쪽 문맥 제한**

```python
import re

text = '가격: $100, 할인: $50, 세금: €20, 합계: $170'

# $ 뒤에 있고(후방), 뒤에 "0"이 두 번 이상 따라오는(전방) 금액
result = re.findall(r'(?<=\$)\d+(?=0{2,})', text)
# 이런 식으로 양쪽 조건을 동시에 적용할 수 있습니다
```

**조합 패턴 2: 여러 전방 탐색을 연결하여 AND 조건 구현**

이것이 전후방 탐색의 가장 강력한 활용법 중 하나입니다. 전방 탐색은 문자를 소비하지 않으므로, **같은 위치에서 여러 전방 탐색을 연속으로 사용**할 수 있습니다. 각 전방 탐색은 독립적으로 조건을 확인하고, 모두 성공해야 전체 매칭이 진행됩니다.

```
위치 ──┬── (?=조건A) → 확인 후 위치 복귀
       ├── (?=조건B) → 확인 후 위치 복귀
       ├── (?=조건C) → 확인 후 위치 복귀
       └── [메인 패턴] → 실제 매칭 수행
```

이는 논리적으로 `조건A AND 조건B AND 조건C AND 메인패턴`과 동일합니다.

### 상세 예시

> 🔍 **예시 9.6.1: 비밀번호 강도 검증**
>
> **상황**: 비밀번호가 다음 조건을 **모두** 만족하는지 검증하고 싶습니다.
> - 최소 8자 이상
> - 대문자 1개 이상 포함
> - 소문자 1개 이상 포함
> - 숫자 1개 이상 포함
>
> ```python
> import re
> 
> pattern = r'^(?=.*[A-Z])(?=.*[a-z])(?=.*\d).{8,}$'
> 
> passwords = ['Abc12345', 'abcdefgh', 'ABCD1234', 'Ab1', 'MyP@ss99']
> 
> for pw in passwords:
>     match = re.fullmatch(pattern, pw)
>     status = '✅ 통과' if match else '❌ 실패'
>     print(f'{pw:15s} → {status}')
> ```
>
> **출력**:
> ```
> Abc12345        → ✅ 통과
> abcdefgh        → ❌ 실패  (대문자, 숫자 없음)
> ABCD1234        → ❌ 실패  (소문자 없음)
> Ab1             → ❌ 실패  (8자 미만)
> MyP@ss99        → ✅ 통과
> ```
>
> **풀이/설명** — 패턴을 분해해봅시다:
>
> ```
> ^                  ← 문자열의 시작
> (?=.*[A-Z])        ← 전방 탐색 #1: 어딘가에 대문자가 있는가?
> (?=.*[a-z])        ← 전방 탐색 #2: 어딘가에 소문자가 있는가?
> (?=.*\d)           ← 전방 탐색 #3: 어딘가에 숫자가 있는가?
> .{8,}              ← 메인 패턴: 8자 이상의 임의 문자
> $                  ← 문자열의 끝
> ```
>
> 세 개의 전방 탐색은 모두 `^`(문자열 시작) 위치에서 실행됩니다. 각각 `.*`로 문자열 전체를 탐색하여 조건을 확인합니다. 전방 탐색이 소비하지 않으므로, 세 번 모두 같은 시작 위치에서 독립적으로 실행됩니다. 세 조건이 모두 통과하면, 비로소 `.{8,}`이 실제 매칭을 수행합니다.
>
> **핵심 포인트**: 전방 탐색 조합 = AND 논리. 이 테크닉은 비밀번호 검증 외에도, "여러 조건을 동시에 만족하는 문자열 찾기"라는 패턴에 범용적으로 활용됩니다.

> 🔍 **예시 9.6.2: 천 단위 쉼표 삽입 — 완성 버전**
>
> **상황**: Section 9.2에서 미완성이었던 천 단위 쉼표 삽입을 후방+전방 탐색 조합으로 완성합니다.
>
> ```python
> import re
> 
> numbers = ['1234567890', '1000', '100', '42', '9876543']
> 
> for num in numbers:
>     result = re.sub(r'(?<=\d)(?=(\d{3})+$)', ',', num)
>     print(f'{num:>12s} → {result}')
> ```
>
> **출력**:
> ```
>   1234567890 → 1,234,567,890
>         1000 → 1,000
>          100 → 100
>           42 → 42
>      9876543 → 9,876,543
> ```
>
> **풀이/설명**:
>
> ```
> (?<=\d)           ← 후방 탐색: 왼쪽에 숫자가 있는 위치
>                      (= 맨 앞이 아닌 위치 → 맨 앞 쉼표 방지!)
> (?=(\d{3})+$)     ← 전방 탐색: 오른쪽에 3자리 숫자 묶음이 하나 이상 있고
>                      그 뒤가 문자열 끝인 위치
> ```
>
> 이 패턴은 "왼쪽에 숫자가 있고, 오른쪽에 정확히 3의 배수 개 숫자가 있는 **위치**"를 찾습니다. 이 위치는 문자 사이의 빈 공간이므로, `re.sub()`으로 그 위치에 `,`를 삽입합니다.
>
> Section 9.2의 미완성 버전(`(?=(\d{3})+$)`)은 맨 앞 위치도 매칭하여 `,1,234,...`가 되었지만, `(?<=\d)`를 추가함으로써 "왼쪽에 숫자가 있어야" 하므로 맨 앞은 자연스럽게 제외됩니다.
>
> **핵심 포인트**: 후방 탐색 + 전방 탐색의 조합으로 "양쪽 문맥을 동시에 충족하는 위치"를 정밀하게 찾을 수 있습니다.

> 🔍 **예시 9.6.3: 문맥 기반 데이터 추출**
>
> **상황**: 설정 텍스트에서 `key = value` 형식의 값 부분만 추출하되, 주석(`#`)으로 시작하는 줄은 제외하고 싶습니다.
>
> ```python
> import re
> 
> config = """
> host = localhost
> port = 8080
> # timeout = 30
> debug = true
> # mode = production
> name = my_app
> """
> 
> # 줄 시작이 #이 아닌 줄에서 = 뒤의 값 추출
> # 여러 줄이므로 각 줄을 개별 처리
> lines = config.strip().split('\n')
> for line in lines:
>     match = re.search(r'(?<!#\s)(?<!#)(\w+)\s*=\s*(\S+)', line.strip())
>     if match and not line.strip().startswith('#'):
>         print(f'{match.group(1)} → {match.group(2)}')
> ```
>
> **출력**:
> ```
> host → localhost
> port → 8080
> debug → true
> name → my_app
> ```
>
> 💬 **참고**: 실무에서는 주석 처리처럼 줄 단위 로직이 필요한 경우, 정규표현식만으로 해결하기보다 파이썬 코드와 결합하는 것이 더 깔끔할 때가 많습니다. 위 예시처럼 `startswith('#')`로 먼저 거르고, 정규표현식으로 값을 추출하는 하이브리드 접근이 실무에서 권장됩니다.

> 🔬 **Deep Dive**: 전방 탐색 AND 패턴의 수학적 의미
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 여러 전방 탐색을 연결하는 `(?=A)(?=B)(?=C)` 패턴이 AND 논리와 동일한 이유를 좀 더 엄밀하게 살펴봅시다.
>
> 전방 탐색 `(?=X)`는 "현재 위치에서 시작하는 부분 문자열이 X에 매칭되는가?"라는 **술어(predicate)**입니다. 이 술어는 참/거짓만 반환하고 상태를 변경하지 않습니다(제로 너비).
>
> 따라서 `(?=A)(?=B)`는:
> 1. 현재 위치에서 A를 확인 → 통과 → 커서를 같은 위치로 복원
> 2. 같은 위치에서 B를 확인 → 통과 → 커서를 같은 위치로 복원
>
> 이것은 논리곱(conjunction) `A ∧ B`와 동일합니다. 정규표현식에는 직접적인 AND 연산자가 없지만, 전방 탐색을 통해 이를 우회적으로 구현할 수 있습니다.
>
> 이 원리를 확장하면, 임의 개수의 조건을 AND로 결합할 수 있습니다:
>
> ```
> (?=조건1)(?=조건2)...(?=조건N)[메인패턴]
> ≡ 조건1 AND 조건2 AND ... AND 조건N AND 메인패턴
> ```
>
> 단, 전방 탐색을 많이 쌓을수록 매칭 엔진의 부하가 증가합니다. 각 전방 탐색마다 문자열을 다시 스캔해야 하기 때문입니다. 이 성능 측면은 Chapter 15와 Chapter 16에서 자세히 다룹니다.

> ⚠️ **주의**: 전후방 탐색 조합 시 흔한 실수들
>
> **실수 1: 전방 탐색의 순서가 결과에 영향을 준다고 오해**
> ```python
> # 이 두 패턴은 동일한 결과를 냅니다
> r'^(?=.*\d)(?=.*[A-Z]).{8,}$'
> r'^(?=.*[A-Z])(?=.*\d).{8,}$'
> ```
> 전방 탐색은 각각 독립적으로 확인하고 커서를 복원하므로, 순서는 결과에 영향을 주지 않습니다 (다만, 성능에는 미미한 영향이 있을 수 있습니다).
>
> **실수 2: 전방 탐색에서 `.*`를 빠뜨리기**
> ```python
> # 잘못: "문자열 시작이 대문자여야 한다"는 의미가 됨
> r'^(?=[A-Z])(?=\d).{8,}$'
> 
> # 올바름: "문자열 어딘가에 대문자가 있어야 한다"
> r'^(?=.*[A-Z])(?=.*\d).{8,}$'
> ```
> `.*`는 "어딘가에"라는 의미를 만드는 핵심 요소입니다.
>
> **실수 3: 메인 매칭 패턴을 빠뜨리기**
> ```python
> # 전방 탐색만 있고 실제 매칭이 없음
> re.fullmatch(r'^(?=.*[A-Z])(?=.*\d)$', 'Abc123')
> # None! — 전방 탐색만으로는 문자를 소비하지 않으므로
> #          ^와 $가 같은 위치가 되어야 함 → 빈 문자열만 매칭
> 
> # 올바름: 메인 매칭 패턴 추가
> re.fullmatch(r'^(?=.*[A-Z])(?=.*\d).+$', 'Abc123')
> # <re.Match object; span=(0, 6), match='Abc123'>
> ```

### Section 요약

- 전방 탐색과 후방 탐색을 조합하면 양쪽 문맥을 동시에 제한할 수 있다
- 여러 전방 탐색을 연결하면 AND 논리를 구현할 수 있다 — 비밀번호 검증의 핵심 기법
- 천 단위 쉼표 삽입처럼 "위치를 찾아 삽입"하는 패턴에 전후방 조합이 효과적이다
- 전방 탐색 AND 패턴에서 `.*`와 메인 매칭 패턴을 빠뜨리지 않도록 주의해야 한다

# Part C: 통합 및 정리 (Integration Block)

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 정규표현식의 고급 기법인 전후방 탐색(lookaround)을 학습했습니다.

1. **전후방 탐색의 본질** (9.1): 전후방 탐색은 제로 너비 단언으로, 문자를 소비하지 않고 주변 문맥만 확인합니다. 이미 배운 `^`, `$`, `\b`의 범용 확장입니다.

2. **긍정 전방 탐색 `(?=...)`** (9.2): 현재 위치의 오른쪽에 특정 패턴이 **있는지** 확인합니다. "뒤에 특정 것이 따라오는 부분만 매칭"하는 데 사용합니다.

3. **부정 전방 탐색 `(?!...)`** (9.3): 오른쪽에 특정 패턴이 **없는지** 확인합니다. 특정 패턴을 배제하는 데 강력합니다.

4. **긍정 후방 탐색 `(?<=...)`** (9.4): 현재 위치의 왼쪽에 특정 패턴이 **있는지** 확인합니다. 파이썬에서는 고정 길이 패턴만 사용 가능합니다.

5. **부정 후방 탐색 `(?<!...)`** (9.5): 왼쪽에 특정 패턴이 **없는지** 확인합니다. 이스케이프되지 않은 문자 찾기 등에 유용합니다.

6. **전후방 탐색 조합** (9.6): 여러 전방 탐색을 연결하면 AND 논리를 구현할 수 있습니다. 비밀번호 검증, 천 단위 쉼표 삽입 등 복합 조건 패턴의 핵심 기법입니다.

## 핵심 용어 정리 (Glossary)

| 용어 | 정의 | 관련 Section |
|------|------|-------------|
| 전방 탐색 (Lookahead) | 현재 위치에서 오른쪽(앞) 방향의 패턴을 확인하는 제로 너비 단언 | 9.1, 9.2, 9.3 |
| 후방 탐색 (Lookbehind) | 현재 위치에서 왼쪽(뒤) 방향의 패턴을 확인하는 제로 너비 단언 | 9.1, 9.4, 9.5 |
| 긍정 전방 탐색 `(?=...)` | 오른쪽에 패턴이 **있을 때** 매칭 성공 | 9.2 |
| 부정 전방 탐색 `(?!...)` | 오른쪽에 패턴이 **없을 때** 매칭 성공 | 9.3 |
| 긍정 후방 탐색 `(?<=...)` | 왼쪽에 패턴이 **있을 때** 매칭 성공 | 9.4 |
| 부정 후방 탐색 `(?<!...)` | 왼쪽에 패턴이 **없을 때** 매칭 성공 | 9.5 |
| 제로 너비 단언 (Zero-width Assertion) | 문자를 소비하지 않고 위치만 확인하는 패턴 요소 | 9.1 |
| 고정 길이 제약 (Fixed-width Constraint) | 파이썬 `re`에서 후방 탐색 패턴의 길이가 사전 결정 가능해야 하는 제약 | 9.4, 9.5 |

## 자기 점검 질문 (Self-Check Questions)

**Q1.** 전방 탐색 `(?=abc)`는 문자 `abc`를 소비한다. (O/X)

<details>
<summary>정답</summary>

**X** — 전방 탐색은 제로 너비 단언으로, 문자를 소비하지 않습니다. 조건만 확인하고 커서를 원래 위치로 복원합니다.
</details>

**Q2.** `re.findall(r'\d+(?=원)', '1000원 2000달러 3000원')`의 결과는?

<details>
<summary>정답</summary>

**`['1000', '3000']`** — `(?=원)`은 "뒤에 `원`이 있는 숫자"만 매칭합니다. `2000` 뒤에는 `달`이 오므로 제외됩니다.
</details>

**Q3.** 파이썬 `re` 모듈에서 `(?<=\d+)` 패턴은 사용할 수 있다. (O/X)

<details>
<summary>정답</summary>

**X** — 파이썬 `re` 모듈의 후방 탐색은 고정 길이 패턴만 허용합니다. `\d+`는 가변 길이이므로 `re.error`가 발생합니다.
</details>

**Q4.** `(?!abc)`는 어떤 종류의 전후방 탐색인가?

<details>
<summary>정답</summary>

**부정 전방 탐색(Negative Lookahead)** — `?!`는 "아닌(부정)" + "전방"을 의미하며, 현재 위치 오른쪽에 `abc`가 없는 위치를 매칭합니다.
</details>

**Q5.** 비밀번호 검증 패턴 `^(?=.*[A-Z])(?=.*\d).{8,}$`에서 `(?=.*[A-Z])`의 역할은?

<details>
<summary>정답</summary>

문자열 시작 위치에서 "어딘가에 대문자(`[A-Z]`)가 하나 이상 존재하는지" 확인합니다. `.*`가 있어야 "어딘가에"라는 의미가 되며, 이 전방 탐색이 통과해야 다음 조건(`(?=.*\d)`)과 메인 패턴(`.{8,}`)이 실행됩니다.
</details>

**Q6.** `re.findall(r'(?<=\$)\d+', '$100 €200 $300')`의 결과는?

<details>
<summary>정답</summary>

**`['100', '300']`** — `(?<=\$)`는 왼쪽에 `$`가 있는 위치에서만 `\d+`를 매칭합니다. `€` 뒤의 `200`은 제외됩니다.
</details>

**Q7.** 전방 탐색 `(?=A)(?=B)`에서 두 탐색의 순서를 바꾸면 결과가 달라진다. (O/X)

<details>
<summary>정답</summary>

**X** — 전방 탐색은 각각 독립적으로 같은 위치에서 실행되므로, 순서를 바꿔도 결과는 동일합니다. (단, 성능에는 미미한 차이가 있을 수 있습니다.)
</details>

## D2. Exercise (응용 문제) — 개념 적용 및 통합

> **Exercise 9.1**: HTML 텍스트에서 `<a href="...">` 태그의 **URL만** 추출하세요 (태그 자체는 제외).
>
> ```python
> html = '<a href="https://example.com">링크1</a> <a href="https://test.org">링크2</a> <img src="logo.png">'
> # 기대 결과: ['https://example.com', 'https://test.org']
> ```
>
> **힌트**: 후방 탐색으로 `<a href="` 뒤를, 전방 탐색으로 `">`앞을 조건으로 사용하세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: `<a href="`는 고정 길이(9글자)이므로 후방 탐색에 사용 가능합니다. URL 부분은 `"`가 아닌 문자들이고, 끝은 `">`으로 표시됩니다.
>
> **풀이 과정**:
> ```python
> import re
> 
> html = '<a href="https://example.com">링크1</a> <a href="https://test.org">링크2</a> <img src="logo.png">'
> 
> result = re.findall(r'(?<=<a href=")[^"]+(?=">)', html)
> print(result)  # ['https://example.com', 'https://test.org']
> ```
>
> **정답**: `['https://example.com', 'https://test.org']`
>
> **보충 설명**: `(?<=<a href=")`는 왼쪽에 `<a href="`가 있는 위치에서 시작하고, `[^"]+`는 `"`가 아닌 문자를 하나 이상 매칭하며, `(?=">)`는 오른쪽에 `">`가 오는지 확인합니다. `<img src="...">`는 후방 탐색 조건(`<a href="`)에 맞지 않으므로 자연스럽게 제외됩니다.
> </details>

> **Exercise 9.2**: 다음 조건을 **모두** 만족하는 사용자 ID인지 검증하는 정규표현식을 작성하세요.
>
> 조건:
> - 4~12자 길이
> - 영문 소문자로 시작
> - 영문 소문자, 숫자, 밑줄(`_`)만 포함
> - 연속된 밑줄(`__`)이 없어야 함
>
> ```python
> # 통과해야 하는 ID: 'user_1', 'abc', 'test_name_01'  (abc는 3자로 실패)
> # 실패해야 하는 ID: '1user', 'ab__cd', 'User', 'a'
> 
> valid_ids = ['user_1', 'abcd', 'test_name01', 'my_id_2']
> invalid_ids = ['1user', 'ab__cd', 'User', 'a', 'very_long_user_name_here']
> ```
>
> **힌트**: 전방 탐색으로 "연속 밑줄 없음" 조건을 AND로 추가하세요.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 기본 구조(`^[a-z][a-z0-9_]{3,11}$`)에 부정 전방 탐색(`(?!.*__)`)을 AND 조건으로 추가합니다.
>
> **풀이 과정**:
> ```python
> import re
> 
> pattern = r'^(?!.*__)[a-z][a-z0-9_]{3,11}$'
> 
> valid_ids = ['user_1', 'abcd', 'test_name01', 'my_id_2']
> invalid_ids = ['1user', 'ab__cd', 'User', 'a', 'very_long_user_name_here']
> 
> print("=== 유효한 ID ===")
> for uid in valid_ids:
>     result = '✅' if re.fullmatch(pattern, uid) else '❌'
>     print(f'  {uid}: {result}')
> 
> print("=== 무효한 ID ===")
> for uid in invalid_ids:
>     result = '✅' if re.fullmatch(pattern, uid) else '❌'
>     print(f'  {uid}: {result}')
> ```
>
> **출력**:
> ```
> === 유효한 ID ===
>   user_1: ✅
>   abcd: ✅
>   test_name01: ✅
>   my_id_2: ✅
> === 무효한 ID ===
>   1user: ❌
>   ab__cd: ❌
>   User: ❌
>   a: ❌
>   very_long_user_name_here: ❌
> ```
>
> **패턴 분해**:
> - `^` — 문자열 시작
> - `(?!.*__)` — 부정 전방 탐색: 어딘가에 `__`가 있으면 실패
> - `[a-z]` — 첫 글자는 소문자
> - `[a-z0-9_]{3,11}` — 나머지 3~11자 (총 4~12자)
> - `$` — 문자열 끝
> </details>

> **Exercise 9.3**: 다음 CSV 데이터에서 쉼표로 분리된 각 필드를 추출하되, **큰따옴표 안의 쉼표는 구분자로 취급하지 않아야** 합니다.
>
> ```python
> line = 'John,"New York, NY",30,"Engineer, Senior"'
> # 기대 결과: ['John', '"New York, NY"', '30', '"Engineer, Senior"']
> ```
>
> **힌트**: 큰따옴표 안의 내용 또는 쉼표가 아닌 문자들을 매칭하는 패턴을 만드세요. 전후방 탐색 외에도 비캡처 그룹을 활용할 수 있습니다.
>
> <details>
> <summary>모범 답안</summary>
>
> **접근 방법**: 각 필드는 "큰따옴표로 감싸진 문자열" 또는 "쉼표가 아닌 문자열"입니다.
>
> **풀이 과정**:
> ```python
> import re
> 
> line = 'John,"New York, NY",30,"Engineer, Senior"'
> 
> # "..."로 감싸진 것 또는 쉼표가 아닌 것
> result = re.findall(r'"[^"]*"|[^,]+', line)
> print(result)
> # ['John', '"New York, NY"', '30', '"Engineer, Senior"']
> ```
>
> **정답**: `['John', '"New York, NY"', '30', '"Engineer, Senior"']`
>
> **보충 설명**: 이 문제는 전후방 탐색 없이도 풀 수 있지만, 전후방 탐색으로 "따옴표 밖의 쉼표" 위치를 찾는 접근도 가능합니다. 실무에서는 CSV 파싱에 전용 라이브러리(`csv` 모듈)를 사용하는 것이 더 안전합니다.
> </details>

## D3. Problem (심화 문제) — 깊이 있는 사고와 창의적 문제 해결

> **Problem 9.1**: 다음 텍스트에서 **한국 전화번호 형식**(010-XXXX-XXXX)의 가운데 4자리만 `****`로 마스킹하세요. 전화번호의 나머지 부분은 유지해야 합니다.
>
> ```python
> text = '연락처: 010-1234-5678, 사무실: 02-987-6543, 핸드폰: 010-9876-5432'
> # 기대 결과: '연락처: 010-****-5678, 사무실: 02-987-6543, 핸드폰: 010-****-5432'
> ```
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 후방 탐색으로 `010-` 뒤를, 전방 탐색으로 `-XXXX` 앞을 조건으로 사용하여, 가운데 4자리만 정밀하게 찾습니다.
>
> **풀이 전략**: `re.sub()`의 패턴에서 후방 탐색(`(?<=010-)`)으로 "010-" 뒤를 확인하고, 4자리 숫자(`\d{4}`)를 매칭하며, 전방 탐색(`(?=-\d{4})`)으로 뒤에 "-숫자4자리"가 오는지 확인합니다.
>
> **상세 풀이**:
> ```python
> import re
> 
> text = '연락처: 010-1234-5678, 사무실: 02-987-6543, 핸드폰: 010-9876-5432'
> 
> result = re.sub(r'(?<=010-)\d{4}(?=-\d{4})', '****', text)
> print(result)
> # 연락처: 010-****-5678, 사무실: 02-987-6543, 핸드폰: 010-****-5432
> ```
>
> **확장 생각**: 이 패턴을 확장하여 앞 3자리도 마스킹하거나, 011/016 등 다른 번호도 처리하려면 후방 탐색을 `(?<=01[016]-)`로 수정할 수 있습니다. 단, 후방 탐색 내의 각 선택지 길이가 동일해야 하는 점에 주의하세요.
> </details>

> **Problem 9.2**: 마크다운 텍스트에서 **인라인 코드(`backtick`)로 감싸지지 않은** URL만 추출하는 패턴을 작성하세요.
>
> ```python
> text = """
> 방문하세요: https://example.com
> 코드에서는 `https://api.example.com`을 사용합니다
> 또한 https://test.org 도 참고하세요
> 인라인 코드: `https://internal.dev`
> """
> # 기대 결과: ['https://example.com', 'https://test.org']
> ```
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 부정 후방 탐색으로 "바로 앞에 `` ` ``이 없는" URL만 매칭합니다.
>
> **풀이 전략**: URL 패턴 앞에 `(?<!` `` ` `` `)`를 배치합니다. 추가로, backtick 안의 URL은 backtick으로 닫히므로, 부정 전방 탐색 없이도 후방 탐색만으로 충분할 수 있습니다.
>
> **상세 풀이**:
> ```python
> import re
> 
> text = """
> 방문하세요: https://example.com
> 코드에서는 `https://api.example.com`을 사용합니다
> 또한 https://test.org 도 참고하세요
> 인라인 코드: `https://internal.dev`
> """
> 
> result = re.findall(r'(?<!`)https?://\S+(?!`)', text)
> # 좀 더 정확하게:
> result = re.findall(r'(?<!`)https?://[^\s`]+', text)
> print(result)
> # ['https://example.com', 'https://test.org']
> ```
>
> **확장 생각**: 실제 마크다운 파싱에서는 backtick이 짝을 이루는지, 코드 블록(triple backtick) 안인지도 고려해야 합니다. 이런 복잡한 구조는 정규표현식만으로 완벽히 처리하기 어려우며, 전용 마크다운 파서와 조합하는 것이 실무에서 더 안전합니다. 이 한계에 대해서는 Chapter 17에서 다룹니다.
> </details>

> **Problem 9.3**: 다음 조건을 모두 만족하는 **강력한 비밀번호 검증 패턴**을 작성하고, 테스트하세요.
>
> 조건:
> 1. 8~20자 길이
> 2. 대문자 1개 이상
> 3. 소문자 1개 이상
> 4. 숫자 1개 이상
> 5. 특수문자(`!@#$%^&*`) 1개 이상
> 6. 같은 문자가 3번 연속 반복되지 않음 (예: `aaa`, `111` 불허)
> 7. 공백 포함 불가
>
> 테스트 케이스:
> ```python
> passwords = [
>     'MyP@ss1!',      # ✅
>     'Str0ng!Pass',   # ✅
>     'weakpassword',  # ❌ (대문자, 숫자, 특수문자 없음)
>     'SHORT1!a',      # ✅ (9자)
>     'Aaaa1!bcde',    # ❌ (aaa 연속)
>     'NoSpecial1',    # ❌ (특수문자 없음)
>     'Has Space1!A',  # ❌ (공백 포함)
>     'A1!a',          # ❌ (8자 미만)
> ]
> ```
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 6개의 조건을 각각 전방 탐색으로 AND 결합합니다. "같은 문자 3연속 금지"는 부정 전방 탐색 `(?!.*(.)(\1){2})`로 표현합니다.
>
> **풀이 전략**:
> ```
> ^                         ← 시작
> (?=.*[A-Z])               ← 대문자 포함
> (?=.*[a-z])               ← 소문자 포함
> (?=.*\d)                  ← 숫자 포함
> (?=.*[!@#$%^&*])          ← 특수문자 포함
> (?!.*(.)\1\1)             ← 같은 문자 3연속 없음
> (?!.*\s)                  ← 공백 없음 (또는 메인 패턴에서 \S 사용)
> .{8,20}                   ← 8~20자
> $                         ← 끝
> ```
>
> **상세 풀이**:
> ```python
> import re
> 
> pattern = r'^(?=.*[A-Z])(?=.*[a-z])(?=.*\d)(?=.*[!@#$%^&*])(?!.*(.)\1\1)\S{8,20}$'
> 
> passwords = [
>     ('MyP@ss1!', True),
>     ('Str0ng!Pass', True),
>     ('weakpassword', False),
>     ('SHORT1!a', True),
>     ('Aaaa1!bcde', False),
>     ('NoSpecial1', False),
>     ('Has Space1!A', False),
>     ('A1!a', False),
> ]
> 
> for pw, expected in passwords:
>     match = re.fullmatch(pattern, pw)
>     actual = bool(match)
>     status = '✅' if actual == expected else '❌ 불일치!'
>     print(f'{pw:20s} → {"통과" if actual else "실패":4s} {status}')
> ```
>
> **확장 생각**: 실제 서비스에서는 정규표현식 하나로 모든 비밀번호 규칙을 검증하기보다, 각 규칙을 별도의 `re.search()`로 확인하고 실패 사유를 개별적으로 사용자에게 안내하는 것이 UX 측면에서 더 좋습니다. 정규표현식은 "통과/실패"만 알려주지, "어떤 조건이 미충족인지"는 알려주지 않기 때문입니다.
> </details>
