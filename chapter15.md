## 15.1 유한 오토마타와 정규표현식

> 💡 **한 줄 요약**: 정규표현식은 수학적으로 "유한 오토마타"라는 기계 모델에 기반하며, 이 이론적 배경이 엔진의 동작 방식을 결정합니다.

### 직관적 이해

지금까지 우리는 정규표현식의 *사용법*에 집중해왔습니다. `\d+`가 숫자를 찾고, `.*`가 모든 문자를 삼킨다는 것은 이미 알고 있습니다. 하지만 한 번도 물어보지 않은 질문이 있습니다. **파이썬은 이 패턴을 어떻게 처리하는가?**

비유를 들어봅시다. 여러분이 미로를 탈출해야 하는 상황을 떠올려 보세요. 미로의 각 갈림길에서 어떤 전략을 쓰느냐에 따라 탈출 속도가 극적으로 달라집니다.

- **전략 A**: 모든 갈림길을 기억해두고, 막다른 길이 나오면 가장 최근 갈림길로 돌아가 다른 길을 시도합니다.
- **전략 B**: 분신술을 써서 갈림길마다 자신의 복제를 보내, 모든 경로를 동시에 탐색합니다.

정규표현식 엔진도 정확히 이 두 가지 전략 중 하나를 사용합니다. 전략 A는 **NFA**(비결정적 유한 오토마타) 엔진이고, 전략 B는 **DFA**(결정적 유한 오토마타) 엔진입니다. 파이썬의 `re` 모듈은 **전략 A, 즉 NFA 방식**을 사용합니다.

이 전략의 차이가 왜 중요한지는 이 Chapter 전체를 통해 확인하게 될 것입니다.

### 핵심 개념 설명

#### 유한 오토마타란?

> 📝 **유한 오토마타(Finite Automaton)**: 유한한 개수의 **상태(state)**를 가지며, 입력 문자를 하나씩 읽으면서 정해진 규칙에 따라 상태를 전이(transition)하는 추상적 기계 모델입니다. 최종적으로 **수락 상태(accept state)**에 도달하면 매칭 성공, 그렇지 않으면 실패입니다.

어렵게 들릴 수 있지만, 핵심은 단순합니다. 유한 오토마타는 다음 세 가지로 구성됩니다:

1. **상태들의 집합**: 기계가 처할 수 있는 모든 "상황"
2. **전이 규칙**: "이 상태에서 이 문자를 읽으면 어느 상태로 간다"는 규칙
3. **수락 상태**: "여기에 도달하면 매칭 성공"이라고 판단하는 특별한 상태

정규표현식 `ab`를 유한 오토마타로 표현하면 이렇게 됩니다:

```
         'a'         'b'
[시작] ──────→ [S1] ──────→ [[수락]]
```

문자열 `"ab"`를 입력하면:
1. [시작] 상태에서 `'a'`를 읽음 → [S1]로 이동
2. [S1] 상태에서 `'b'`를 읽음 → [[수락]]으로 이동 → **매칭 성공!**

문자열 `"ac"`를 입력하면:
1. [시작] 상태에서 `'a'`를 읽음 → [S1]로 이동
2. [S1] 상태에서 `'c'`를 읽음 → 전이 규칙 없음 → **매칭 실패!**

#### NFA와 DFA의 핵심 차이

> 📝 **DFA(Deterministic Finite Automaton, 결정적 유한 오토마타)**: 각 상태에서 특정 입력 문자에 대해 이동할 수 있는 다음 상태가 **정확히 하나**뿐인 오토마타입니다. 선택의 여지가 없으므로 "결정적"이라 합니다.

> 📝 **NFA(Nondeterministic Finite Automaton, 비결정적 유한 오토마타)**: 각 상태에서 같은 입력 문자에 대해 **여러 개의 다음 상태**로 이동할 수 있는 오토마타입니다. 또한, 입력 문자를 소비하지 않고 상태를 전이하는 **엡실론 전이(ε-transition)**도 가능합니다.

패턴 `a|b`를 각 방식으로 표현해보면 차이가 명확해집니다:

**DFA 방식** (각 상태에서 경로가 하나):
```
         'a'
[시작] ──────→ [[수락]]
   │
   │     'b'
   └──────────→ [[수락]]
```

**NFA 방식** (엡실론 전이로 분기가 생김):
```
            ε         'a'
[시작] ──────→ [S1] ──────→ [[수락]]
   │
   │        ε         'b'
   └──────→ [S2] ──────→ [[수락]]
```

NFA에서 `ε`(엡실론)은 "문자를 읽지 않고도 이동할 수 있다"는 뜻입니다. [시작] 상태에서 엔진은 문자를 읽기도 전에 [S1]과 [S2] 두 갈래로 **갈림길**에 서게 됩니다. 이 갈림길을 어떻게 처리하느냐가 NFA 엔진의 핵심이며, 그 답이 바로 **백트래킹**입니다.

#### 왜 파이썬은 NFA를 사용하는가?

파이썬뿐 아니라 Perl, Java, JavaScript, Ruby, .NET 등 대부분의 프로그래밍 언어가 NFA 엔진을 채택합니다. 그 이유는 명확합니다:

| 특성 | NFA | DFA |
|------|-----|-----|
| **캡처 그룹** 지원 | ✅ 자연스럽게 지원 | ❌ 원래 지원 불가 |
| **역참조** 지원 | ✅ 지원 | ❌ 지원 불가 |
| **전후방 탐색** 지원 | ✅ 지원 | ❌ 지원 불가 |
| **탐욕적/비탐욕적 제어** | ✅ 가능 | ❌ 개념 없음 |
| 성능 예측 가능성 | 패턴에 따라 편차 큼 | 항상 입력 길이에 비례 |

우리가 Chapter 6~9에서 배운 캡처 그룹, 역참조, 전후방 탐색 등의 강력한 기능은 모두 NFA 엔진이기에 가능한 것입니다. DFA는 성능이 안정적이지만 이런 기능을 제공할 수 없습니다. 대부분의 프로그래밍 언어가 기능의 풍부함을 선택한 것입니다.

> 🔗 **연결**: Chapter 6에서 배운 캡처 그룹 `()`, Chapter 8의 명명 그룹과 역참조, Chapter 9의 전후방 탐색은 모두 NFA 엔진의 특성 덕분에 가능합니다.

### 상세 예시

> 🔍 **예시 15.1.1: 단순 패턴의 오토마타 표현**
>
> **상황**: 패턴 `\d{3}`(숫자 3자리)를 오토마타로 표현해봅시다.
>
> **풀이/설명**:
> ```
>          \d          \d          \d
> [시작] ──────→ [S1] ──────→ [S2] ──────→ [[수락]]
> ```
>
> 문자열 `"123abc"`에 대해:
> 1. [시작]에서 `'1'`(숫자) → [S1]로 이동
> 2. [S1]에서 `'2'`(숫자) → [S2]로 이동
> 3. [S2]에서 `'3'`(숫자) → [[수락]] → **매칭 성공: `"123"`**
>
> 문자열 `"12a"`에 대해:
> 1. [시작]에서 `'1'`(숫자) → [S1]
> 2. [S1]에서 `'2'`(숫자) → [S2]
> 3. [S2]에서 `'a'`(숫자 아님) → 전이 불가 → **매칭 실패**
>
> **핵심 포인트**: 오토마타는 문자를 하나씩 읽으며 상태를 이동합니다. 전이 규칙에 맞지 않는 문자를 만나면 그 즉시 실패합니다.

> 🔍 **예시 15.1.2: 선택이 있는 패턴의 NFA**
>
> **상황**: 패턴 `cat|car`를 NFA로 표현해봅시다.
>
> **풀이/설명**:
> ```
>            ε       'c'       'a'       't'
> [시작] ──────→ [A1] ───→ [A2] ───→ [A3] ───→ [[수락]]
>    │
>    │      ε       'c'       'a'       'r'
>    └──────→ [B1] ───→ [B2] ───→ [B3] ───→ [[수락]]
> ```
>
> 문자열 `"car"`에 대해 NFA 엔진의 동작:
> 1. [시작]에서 ε-전이로 [A1] 경로를 **먼저 선택** (위쪽 분기 우선)
> 2. [A1]에서 `'c'` → [A2], [A2]에서 `'a'` → [A3]
> 3. [A3]에서 `'r'`을 읽어야 하는데, `'t'`만 전이 가능 → **실패**
> 4. 🔙 **백트래킹**: [시작]으로 돌아가서 [B1] 경로를 시도
> 5. [B1]에서 `'c'` → [B2], [B2]에서 `'a'` → [B3], [B3]에서 `'r'` → [[수락]]
> 6. **매칭 성공: `"car"`**
>
> **핵심 포인트**: NFA는 갈림길에서 한 경로를 먼저 시도하고, 실패하면 돌아와서 다른 경로를 시도합니다. 이 "돌아오는" 과정이 바로 **백트래킹**입니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: 정규표현식의 수학적 기원**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 정규표현식의 역사는 1950년대 수학자 **스티븐 클리니(Stephen Kleene)**로 거슬러 올라갑니다. 그는 신경학자 워렌 매컬록과 월터 피츠가 제안한 신경망 모델을 연구하다가, **정규 집합(regular set)**을 기술하기 위한 표기법을 개발했습니다. 이것이 정규표현식의 원형입니다.
>
> 수학적으로 중요한 정리가 하나 있습니다: **클리니의 정리(Kleene's Theorem)**는 "정규표현식으로 표현할 수 있는 언어의 집합"과 "유한 오토마타가 인식할 수 있는 언어의 집합"이 **동치**라고 말합니다. 즉, 정규표현식과 유한 오토마타는 같은 능력을 가진 서로 다른 표현법입니다.
>
> 다만 현대의 프로그래밍 언어에서 사용하는 정규표현식(역참조, 전후방 탐색 등)은 원래의 수학적 정규표현식을 **넘어서는** 기능을 가지고 있습니다. 엄밀히 말하면, 역참조가 포함된 패턴은 더 이상 "정규(regular)" 언어가 아니지만, 역사적 이유로 "정규표현식"이라는 이름을 계속 사용하고 있습니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: "정규표현식 = 유한 오토마타"라고 단순히 등치하면 안 됩니다. 수학적 의미의 정규표현식과 프로그래밍에서의 정규표현식은 범위가 다릅니다. 파이썬의 `re` 모듈이 제공하는 역참조(`\1`), 전후방 탐색 등은 이론적 정규표현식의 범위를 넘어서며, 이것이 NFA 엔진의 백트래킹이 필수적인 이유이기도 합니다.

### Section 요약

- 정규표현식은 수학적으로 **유한 오토마타**라는 기계 모델에 기반합니다
- **DFA**는 각 상태에서 다음 상태가 하나로 결정되며, **NFA**는 여러 가능성을 가질 수 있습니다
- 파이썬의 `re` 모듈은 **NFA 기반 엔진**을 사용하며, 이는 캡처 그룹, 역참조, 전후방 탐색 등을 지원하기 위함입니다
- NFA 엔진은 갈림길에서 한 경로를 먼저 시도하고, 실패하면 되돌아가는 **백트래킹** 방식으로 동작합니다

## 15.2 NFA 기반 매칭의 동작

> 💡 **한 줄 요약**: 파이썬의 NFA 엔진은 패턴의 각 요소를 왼쪽에서 오른쪽으로 시도하며, 분기점마다 상태를 저장해두고 실패 시 되돌아가는 방식으로 매칭합니다.

### 직관적 이해

NFA 엔진의 매칭 과정을 이해하려면, 여러분이 **책을 읽는 탐정**이라고 상상해보세요.

여러분 앞에 용의자 목록(문자열)이 있고, 프로파일(패턴)에 맞는 범인을 찾아야 합니다. 프로파일의 조건은 왼쪽부터 순서대로 확인합니다. 만약 어떤 조건에서 "이것일 수도 있고 저것일 수도 있다"는 상황이 생기면, **책갈피(bookmark)**를 꽂아두고 한쪽부터 추적합니다. 추적이 막다른 길에 이르면, 책갈피가 꽂힌 곳으로 돌아와 다른 가능성을 추적합니다.

이 "책갈피"가 바로 NFA 엔진이 저장하는 **백트래킹 상태**이며, "돌아오는" 행위가 **백트래킹**입니다.

### 핵심 개념 설명

#### NFA 매칭의 기본 원칙

파이썬의 NFA 엔진은 다음 규칙을 따릅니다:

**원칙 1: 패턴 주도(Pattern-driven)**

엔진은 패턴의 각 요소를 **왼쪽에서 오른쪽**으로 하나씩 처리합니다. 각 패턴 요소에 대해 현재 입력 위치의 문자가 일치하는지 확인하고, 일치하면 다음 패턴 요소로 넘어갑니다.

```python
# 패턴: abc
# 문자열: "abcdef"

# 엔진의 처리:
# 패턴 'a' → 입력 'a' ✅ → 다음으로
# 패턴 'b' → 입력 'b' ✅ → 다음으로
# 패턴 'c' → 입력 'c' ✅ → 패턴 끝 → 매칭 성공!
```

**원칙 2: 분기점에서 상태 저장**

선택(`|`), 수량자(`*`, `+`, `?`) 등에서 여러 가능성이 있으면, 엔진은 현재 상태(패턴 위치 + 입력 위치)를 **저장**해두고 한 가지를 먼저 시도합니다.

**원칙 3: 실패 시 백트래킹**

현재 시도가 실패하면, 가장 최근에 저장한 상태로 돌아가서 다른 선택지를 시도합니다. 모든 저장된 상태가 소진되면 최종적으로 매칭 실패입니다.

**원칙 4: 입력 위치 이동과 재시도**

`re.search()`의 경우, 문자열의 위치 0에서 시작하여 패턴 매칭을 시도합니다. 실패하면 위치 1, 위치 2, ... 순으로 시작점을 이동하며 반복합니다. 문자열 끝까지 모두 실패하면 최종 실패(`None` 반환)입니다.

```python
import re

# re.search(r'xyz', 'abcxyz')의 내부 동작:
# 위치 0 ('a'부터): 'x' != 'a' → 실패, 위치 이동
# 위치 1 ('b'부터): 'x' != 'b' → 실패, 위치 이동
# 위치 2 ('c'부터): 'x' != 'c' → 실패, 위치 이동
# 위치 3 ('x'부터): 'x' == 'x' ✅, 'y' == 'y' ✅, 'z' == 'z' ✅ → 매칭 성공!
```

> 💬 **참고**: `re.match()`는 항상 위치 0에서만 시도하므로 이 재시도 과정이 없습니다. `re.search()`만 시작 위치를 이동하며 반복합니다.

#### 선택(`|`)에서의 NFA 동작

> 🔗 **연결**: Chapter 2에서 배운 선택 연산자 `|`의 동작을 엔진 수준에서 살펴봅니다.

선택 연산자를 만나면 NFA 엔진은 **왼쪽 선택지를 먼저** 시도합니다. 이것은 매우 중요한 규칙입니다:

```python
import re

# 패턴 'cat|catch'에서:
# NFA는 'cat'을 먼저 시도합니다

result = re.search(r'cat|catch', 'catch a cat')
print(result.group())  # 'cat' — 'catch'가 아니라 'cat'!
```

왜 `'catch'`가 아니라 `'cat'`이 매칭될까요? 엔진의 동작을 추적해봅시다:

```
입력: "catch a cat"
패턴: cat|catch

위치 0에서 시작:
  선택지 1 'cat' 시도:
    'c' == 'c' ✅, 'a' == 'a' ✅, 't' == 't' ✅
    → 'cat' 매칭 성공! (패턴의 첫 번째 선택지가 성공했으므로 여기서 멈춤)
```

NFA 엔진은 첫 번째 성공하는 선택지를 찾으면 **즉시 매칭을 확정**합니다. 두 번째 선택지가 더 길게 매칭될 수 있더라도 시도하지 않습니다. 이를 **"왼쪽 우선(leftmost)"** 규칙이라 합니다.

#### 수량자에서의 NFA 동작

> 🔗 **연결**: Chapter 4에서 배운 탐욕적/비탐욕적 매칭의 원리가 NFA 엔진의 동작 방식에서 비롯됩니다.

수량자를 만나면 엔진은 **"얼마나 많이 반복할 것인가"**에 대해 선택의 분기가 생깁니다:

- **탐욕적 수량자** (`*`, `+`, `?`): "가능한 한 많이" 매칭을 먼저 시도하고, 실패 시 하나씩 줄여감
- **비탐욕적 수량자** (`*?`, `+?`, `??`): "가능한 한 적게" 매칭을 먼저 시도하고, 실패 시 하나씩 늘려감

이것은 다음 Section에서 더 자세히 다룹니다. 여기서는 수량자가 NFA 엔진에서 **분기점을 생성한다**는 핵심만 기억해두세요.

### 상세 예시

> 🔍 **예시 15.2.1: NFA 매칭의 단계별 추적 — 리터럴 패턴**
>
> **상황**: `re.search(r'world', 'hello world')`의 내부 동작을 추적합니다.
>
> **풀이/설명**:
> ```
> 입력: "hello world"
>        0123456789...
>
> 패턴: world
>
> [위치 0] 'h': 'w' != 'h' → 실패, 위치 이동
> [위치 1] 'e': 'w' != 'e' → 실패, 위치 이동
> [위치 2] 'l': 'w' != 'l' → 실패, 위치 이동
> [위치 3] 'l': 'w' != 'l' → 실패, 위치 이동
> [위치 4] 'o': 'w' != 'o' → 실패, 위치 이동
> [위치 5] ' ': 'w' != ' ' → 실패, 위치 이동
> [위치 6] 'w': 'w' == 'w' ✅
>   → 'o' == 'o' ✅ → 'r' == 'r' ✅ → 'l' == 'l' ✅ → 'd' == 'd' ✅
>   → 패턴 끝 도달 → 매칭 성공! span=(6, 11)
> ```
>
> **핵심 포인트**: `re.search()`는 문자열의 모든 위치에서 패턴 매칭을 시도합니다. 이 단순한 리터럴 패턴에서도 엔진은 위치 0부터 6까지 7번의 시작점 시도를 수행했습니다.

> 🔍 **예시 15.2.2: 선택에서의 백트래킹**
>
> **상황**: `re.search(r'ab|ac', 'ac')`의 내부 동작을 추적합니다.
>
> **풀이/설명**:
> ```
> 입력: "ac"
> 패턴: ab|ac
>
> [위치 0]
>   선택지 1: 'ab' 시도
>     'a' == 'a' ✅
>     'b' != 'c' ❌ → 선택지 1 실패
>   🔙 백트래킹: 위치 0으로 돌아감
>   선택지 2: 'ac' 시도
>     'a' == 'a' ✅
>     'c' == 'c' ✅
>     → 매칭 성공! "ac"
> ```
>
> 주목할 점은, 선택지 1에서 `'a'`가 이미 성공적으로 매칭되었지만, `'b'`에서 실패하자 엔진이 **입력 위치를 0으로 되돌렸다**는 것입니다. 선택지 2에서 `'a'`를 **다시** 매칭해야 했습니다. 이것이 백트래킹의 비용입니다.
>
> **핵심 포인트**: 백트래킹은 이미 소비한 입력 문자를 "되돌려주는" 과정을 포함합니다. 이는 같은 문자를 여러 번 비교하게 만들 수 있으며, 이것이 성능 문제의 근원입니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: 선택(`|`)의 순서는 결과에 영향을 미칩니다. `cat|catch`와 `catch|cat`은 다른 결과를 낼 수 있습니다. 더 긴 패턴이나 더 구체적인 패턴을 **앞에** 배치해야 의도한 결과를 얻을 수 있습니다.
>
> ```python
> import re
> 
> # 'cat'이 먼저 → 'catch'의 'cat' 부분만 매칭
> re.search(r'cat|catch', 'catch').group()  # 'cat'
> 
> # 'catch'가 먼저 → 전체 'catch' 매칭
> re.search(r'catch|cat', 'catch').group()  # 'catch'
> ```

> ⚠️ **주의**: NFA 엔진은 **"첫 번째 성공"**을 반환합니다. "가장 긴 매칭"이나 "가장 좋은 매칭"을 반환하는 것이 아닙니다. 이 점은 DFA 엔진(항상 가장 긴 매칭 반환)과의 핵심적인 차이입니다.

### Section 요약

- NFA 엔진은 **패턴 주도(pattern-driven)** 방식으로, 패턴의 각 요소를 왼쪽에서 오른쪽으로 처리합니다
- 분기점(선택, 수량자)에서 **상태를 저장**하고 한 경로를 먼저 시도합니다
- 실패 시 **가장 최근 저장점으로 백트래킹**하여 다른 경로를 시도합니다
- 선택(`|`)에서는 **왼쪽 선택지가 우선**이며, 첫 번째 성공 시 즉시 매칭을 확정합니다
- `re.search()`는 문자열의 각 위치에서 순차적으로 매칭을 시도합니다

## 15.3 백트래킹 심화

> 💡 **한 줄 요약**: 백트래킹은 NFA 엔진이 매칭 실패 시 이전 분기점으로 되돌아가 다른 경로를 시도하는 메커니즘이며, 정규표현식의 성능을 결정짓는 핵심 요소입니다.

### 직관적 이해

Chapter 4에서 백트래킹의 기초를 배웠을 때, 우리는 `.*`이 "너무 많이 삼키고 토해내는" 동작을 확인했습니다. 이번에는 그 "토해내는" 과정을 **한 글자 한 글자** 수준에서 정밀하게 추적합니다.

> 🔗 **연결**: Chapter 4 Section 4.5에서 "백트래킹 기초 개념"을 다루었습니다. 이번 Section은 그 기초 위에 상세한 동작 과정을 쌓아올립니다.

미로 탐험 비유를 확장해봅시다. 미로의 각 갈림길에서:

1. 갈림길 위치를 메모장에 기록합니다 (= **상태 저장**)
2. 한 방향으로 들어갑니다 (= **한 경로 시도**)
3. 막다른 길을 만나면 메모장의 마지막 기록을 보고 되돌아갑니다 (= **백트래킹**)
4. 다른 방향을 시도합니다 (= **대안 경로 시도**)
5. 메모장이 비었는데도 출구를 못 찾으면 포기합니다 (= **매칭 실패**)

핵심은 이 메모장(= **백트래킹 스택**)의 크기입니다. 갈림길이 많으면 메모장이 두꺼워지고, 되돌아가는 횟수도 늘어납니다. 이것이 성능 문제의 본질입니다.

### 핵심 개념 설명

#### 백트래킹 스택

> 📝 **백트래킹 스택(Backtracking Stack)**: NFA 엔진이 분기점에서 "나중에 시도할 대안"을 저장하는 내부 자료구조입니다. 스택(후입선출, LIFO) 방식으로 관리되어, 가장 최근에 저장한 대안을 가장 먼저 시도합니다.

백트래킹 스택에 저장되는 정보는:
- **패턴 내 현재 위치**: 어떤 패턴 요소를 시도하고 있었는지
- **입력 문자열 내 현재 위치**: 어디까지 읽었는지
- **캡처 그룹의 현재 상태**: 그룹에 무엇이 캡처되어 있었는지

#### 수량자에서의 백트래킹 상세 추적

탐욕적 수량자 `*`의 백트래킹을 한 글자 수준에서 추적해봅시다.

```python
import re

# 패턴: a.*b
# 문자열: "aXbYb"
result = re.search(r'a.*b', 'aXbYb')
```

엔진의 동작을 단계별로 따라갑니다:

```
입력: "aXbYb"
       01234

패턴: a  .*  b
      ↑

=== 단계 1: 'a' 매칭 ===
위치 0: 'a' == 'a' ✅
패턴 진행 → .* 로 이동

=== 단계 2: .* (탐욕적) — 가능한 한 많이 소비 ===
.* 은 탐욕적이므로 먼저 최대한 많이 삼킵니다:
  위치 1: '.' 매칭 'X' ✅ (계속)
  위치 2: '.' 매칭 'b' ✅ (계속)
  위치 3: '.' 매칭 'Y' ✅ (계속)
  위치 4: '.' 매칭 'b' ✅ (계속)
  위치 5: 문자열 끝 → .* 가 더 이상 매칭 불가, 여기서 멈춤

.* 는 "XbYb" (4글자)를 삼켰습니다.
현재 입력 위치: 5 (문자열 끝)

=== 단계 3: 'b' 매칭 시도 ===
위치 5: 문자열 끝이라 읽을 문자가 없음 ❌

=== 단계 4: 백트래킹 — .* 가 1글자 토해냄 ===
.* 가 마지막으로 삼킨 'b'(위치 4)를 토해냅니다.
.* 는 이제 "XbY" (3글자)를 삼킨 상태.
현재 입력 위치: 4

=== 단계 5: 'b' 매칭 재시도 ===
위치 4: 'b' == 'b' ✅

=== 매칭 성공! ===
결과: "aXbYb" (위치 0~4, 전체 문자열)
```

여기서 핵심적인 관찰을 합시다:

1. `.*`가 `"XbYb"` 전체를 삼킨 후, 뒤의 `b`가 매칭할 문자가 없어서 실패했습니다.
2. `.*`가 한 글자(`'b'`)를 토해내자, 뒤의 `b`가 그 `'b'`와 매칭에 성공했습니다.
3. 결과적으로 `.*`는 `"XbY"`를 매칭하고, 마지막 `b`가 `"b"`(위치 4)와 매칭되었습니다.

문자열 중간의 `'b'`(위치 2)가 아닌 **마지막** `'b'`(위치 4)와 매칭된 것에 주목하세요. 탐욕적 수량자이기 때문에 `.*`가 최대한 많이 삼킨 상태에서 시작하므로, `b`는 가장 뒤의 `'b'`와 매칭됩니다.

#### 비탐욕적 수량자의 백트래킹

같은 상황을 비탐욕적 수량자로 바꿔봅시다:

```python
import re

# 패턴: a.*?b
# 문자열: "aXbYb"
result = re.search(r'a.*?b', 'aXbYb')
```

```
입력: "aXbYb"
       01234

패턴: a  .*?  b
      ↑

=== 단계 1: 'a' 매칭 ===
위치 0: 'a' == 'a' ✅
패턴 진행 → .*? 로 이동

=== 단계 2: .*? (비탐욕적) — 가능한 한 적게 소비 ===
.*? 은 비탐욕적이므로 먼저 0글자 매칭을 시도합니다.
.*? 는 아무것도 삼키지 않음 (0글자).
현재 입력 위치: 1

=== 단계 3: 'b' 매칭 시도 ===
위치 1: 'b' != 'X' ❌

=== 단계 4: 백트래킹 — .*? 가 1글자 더 삼킴 ===
.*? 가 'X'(위치 1)를 삼킵니다.
.*? 는 이제 "X" (1글자).
현재 입력 위치: 2

=== 단계 5: 'b' 매칭 재시도 ===
위치 2: 'b' == 'b' ✅

=== 매칭 성공! ===
결과: "aXb" (위치 0~2)
```

비탐욕적 수량자는 **반대 방향으로 백트래킹**합니다:
- 탐욕적: 최대로 삼킨 후 → 토해내며 백트래킹 (많음 → 적음)
- 비탐욕적: 최소로 삼킨 후 → 더 삼키며 백트래킹 (적음 → 많음)

#### 중첩된 수량자의 백트래킹

수량자가 중첩되면 백트래킹의 복잡도가 폭발적으로 증가할 수 있습니다. 이것이 성능 문제의 가장 위험한 원인입니다.

```python
import re

# 위험한 패턴 예시 (이해를 위한 것이며, 실제로 사용하면 안 됩니다)
# 패턴: (a+)+b
# 문자열: "aaaa" (끝에 'b'가 없음)
```

이 경우 어떤 일이 벌어지는지 생각해봅시다:

- 바깥쪽 `(...)+`는 "그룹을 1회 이상 반복"
- 안쪽 `a+`는 "a를 1회 이상 반복"

`"aaaa"`에서 `'b'`를 찾을 수 없으므로 실패할 것인데, 엔진이 실패를 확정하기 전에 **모든 가능한 분배 방식**을 시도합니다:

```
시도 1: (aaaa) + b  → 'b' 없음 ❌
시도 2: (aaa)(a) + b  → 'b' 없음 ❌
시도 3: (aa)(aa) + b  → 'b' 없음 ❌
시도 4: (aa)(a)(a) + b  → 'b' 없음 ❌
시도 5: (a)(aaa) + b  → 'b' 없음 ❌
시도 6: (a)(aa)(a) + b  → 'b' 없음 ❌
시도 7: (a)(a)(aa) + b  → 'b' 없음 ❌
시도 8: (a)(a)(a)(a) + b  → 'b' 없음 ❌
... (더 많은 조합)
```

`"a"` 4글자에 대해서도 이렇게 많은 시도가 필요합니다. `"a"` 20글자로 늘리면 시도 횟수는 **2^20 = 약 100만 번**으로 폭발합니다! 이것이 **재앙적 백트래킹(catastrophic backtracking)**이며, Chapter 16에서 본격적으로 다루게 됩니다.

> 💬 **참고**: 재앙적 백트래킹의 식별과 해결은 Chapter 16에서 자세히 다룹니다. 이번 Section에서는 그 원인이 되는 메커니즘을 이해하는 것에 집중합니다.

### 상세 예시

> 🔍 **예시 15.3.1: 탐욕적 수량자의 백트래킹 완전 추적**
>
> **상황**: `re.search(r'".*"', 'She said "hello" and "bye"')` — 큰따옴표로 둘러싸인 문자열 추출
>
> **풀이/설명**:
> ```
> 입력: She said "hello" and "bye"
>       0123456789...
>
> 패턴: ".*"
>
> [위치 0~8] '"' != 'S','h','e',' ','s','a','i','d' → 모두 실패, 위치 이동
>
> [위치 9] '"' == '"' ✅
>   .* (탐욕적): 가능한 한 많이 삼킴
>     위치 10: 'h' ✅
>     위치 11: 'e' ✅
>     위치 12: 'l' ✅
>     위치 13: 'l' ✅
>     위치 14: 'o' ✅
>     위치 15: '"' ✅ (. 은 따옴표도 매칭!)
>     위치 16: ' ' ✅
>     위치 17: 'a' ✅
>     위치 18: 'n' ✅
>     위치 19: 'd' ✅
>     위치 20: ' ' ✅
>     위치 21: '"' ✅
>     위치 22: 'b' ✅
>     위치 23: 'y' ✅
>     위치 24: 'e' ✅
>     위치 25: '"' ✅
>     위치 26: 문자열 끝 → .* 멈춤
>
>   닫는 '"' 매칭 시도:
>     위치 26: 문자열 끝 ❌
>   🔙 백트래킹: .* 가 위치 25의 '"'를 토해냄
>     위치 25: '"' == '"' ✅ → 매칭 성공!
>
> 결과: '"hello" and "bye"' (위치 9~25)
> ```
>
> **핵심 포인트**: 탐욕적 `.*`는 두 번째 따옴표까지 삼켜버렸기 때문에, 결과가 `"hello"`가 아니라 `"hello" and "bye"` 전체가 됩니다. 첫 번째 따옴표만 캡처하려면 비탐욕적(`.*?`)이나 부정 문자 클래스(`[^"]*`)를 사용해야 합니다.

> 🔍 **예시 15.3.2: 여러 분기점의 백트래킹 순서**
>
> **상황**: `re.search(r'a(b|c)d', 'acd')` — 선택과 리터럴의 결합
>
> **풀이/설명**:
> ```
> 입력: "acd"
> 패턴: a(b|c)d
>
> [위치 0]
>   'a' == 'a' ✅
>   그룹 (b|c):
>     선택지 1 'b' 시도:
>       위치 1: 'b' != 'c' ❌
>     🔙 백트래킹: 위치 1로 돌아감
>     선택지 2 'c' 시도:
>       위치 1: 'c' == 'c' ✅
>   'd' 매칭:
>     위치 2: 'd' == 'd' ✅
>   → 매칭 성공! "acd"
> ```
>
> **핵심 포인트**: 그룹 내 선택에서도 왼쪽 우선 규칙이 적용됩니다. `'b'`가 먼저 시도되고 실패한 후, 백트래킹을 통해 `'c'`가 시도됩니다.

> 🔍 **예시 15.3.3: 백트래킹 횟수 비교 — 효율적 패턴과 비효율적 패턴**
>
> **상황**: 큰따옴표 안의 내용을 추출하는 두 가지 패턴의 백트래킹 횟수를 비교합니다.
>
> ```python
> import re
> 
> text = '"hello"'
> 
> # 패턴 1: 탐욕적 .* 사용
> # ".*"
> 
> # 패턴 2: 부정 문자 클래스 사용
> # "[^"]*"
> ```
>
> **풀이/설명**:
>
> **패턴 1 (`".*"`)의 동작**:
> ```
> '"' 매칭 ✅
> .* 삼킴: h, e, l, l, o, " (6글자 — 닫는 따옴표까지 삼킴!)
> '"' 매칭: 문자열 끝 ❌
> 🔙 백트래킹: .* 가 '"'를 토해냄
> '"' 매칭: '"' ✅ → 성공
> 
> 총 백트래킹: 1회
> ```
>
> **패턴 2 (`"[^"]*"`)의 동작**:
> ```
> '"' 매칭 ✅
> [^"]* 삼킴: h, e, l, l, o (5글자 — '"'는 [^"]에 매칭 안 됨!)
> '"' 매칭: '"' ✅ → 성공
> 
> 총 백트래킹: 0회
> ```
>
> **핵심 포인트**: `[^"]*`는 따옴표를 애초에 삼키지 않으므로, 닫는 따옴표가 자연스럽게 매칭됩니다. 백트래킹이 전혀 발생하지 않아 더 효율적입니다. 이것이 "구체적인 문자 클래스를 사용하라"는 최적화 원칙의 엔진 수준 근거입니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: 백트래킹의 상태 공간 분석**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 백트래킹의 비용을 정량적으로 이해하기 위해 **상태 공간(state space)**의 개념을 생각해봅시다.
>
> 입력 문자열의 길이가 `n`일 때:
>
> | 패턴 유형 | 상태 공간 크기 | 예시 |
> |-----------|----------------|------|
> | 리터럴 (`abc`) | O(n) | 각 위치에서 한 번씩 비교 |
> | 단순 수량자 (`a*b`) | O(n) | 수량자가 한 번만 백트래킹 |
> | 선택 (`ab\|ac\|ad`) | O(n × 선택지 수) | 각 위치에서 모든 선택지 시도 |
> | **중첩 수량자** (`(a+)+b`) | O(2^n) | 지수적 폭발! |
>
> 중첩 수량자의 O(2^n)은 n=25에서 약 3천만 번, n=30에서 약 10억 번의 시도를 의미합니다. 이것이 정규표현식이 "멈춘 것처럼" 보이는 이유입니다.
>
> 이 분석은 Chapter 16의 성능 최적화에서 구체적인 개선 전략을 세우는 기반이 됩니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: "백트래킹 = 나쁜 것"이라고 오해하지 마세요. 백트래킹은 NFA 엔진의 **정상적인 동작 메커니즘**입니다. 문제가 되는 것은 **과도한** 백트래킹이며, 이는 패턴 설계의 문제입니다. 적절한 수준의 백트래킹은 정규표현식의 유연성을 가능하게 하는 필수 요소입니다.

> ⚠️ **주의**: 탐욕적 수량자 뒤에 같은 문자를 매칭하는 패턴이 오면 불필요한 백트래킹이 발생합니다. 예를 들어 `.*:`보다 `[^:]*:`가 훨씬 효율적입니다. 이 원칙은 "수량자가 삼킬 수 있는 범위를 최대한 구체적으로 제한하라"로 요약됩니다.

### Section 요약

- 백트래킹 스택은 NFA 엔진이 분기점의 대안을 저장하는 **후입선출(LIFO)** 자료구조입니다
- **탐욕적 수량자**는 최대로 삼킨 후 토해내며 백트래킹하고, **비탐욕적 수량자**는 최소로 삼킨 후 더 삼키며 백트래킹합니다
- 백트래킹의 비용은 분기점의 수와 입력 길이에 의해 결정되며, **중첩 수량자**는 지수적 폭발을 일으킬 수 있습니다
- **구체적 문자 클래스**(예: `[^"]*`)를 사용하면 불필요한 백트래킹을 원천 차단할 수 있습니다

## 15.4 탐욕적/비탐욕적 매칭의 엔진 수준 이해

> 💡 **한 줄 요약**: 탐욕적 매칭과 비탐욕적 매칭의 차이는 결국 "분기점에서 어떤 선택을 먼저 시도하느냐"에 불과하며, 이 순서가 매칭 결과와 성능을 좌우합니다.

### 직관적 이해

> 🔗 **연결**: Chapter 4 Section 4.3에서 탐욕적/비탐욕적 매칭의 결과 차이를 배웠습니다. 이번 Section에서는 엔진 내부에서 **왜** 그런 차이가 나는지를 이해합니다.

Chapter 4에서 우리는 탐욕적 수량자가 "가능한 한 많이", 비탐욕적 수량자가 "가능한 한 적게" 매칭한다고 배웠습니다. 하지만 이것은 **결과**에 대한 설명이었을 뿐, **과정**에 대한 설명은 아니었습니다.

엔진의 시각에서 보면, 탐욕적과 비탐욕적의 차이는 놀라울 만큼 단순합니다:

- **탐욕적**: "한 번 더 매칭하기"를 먼저 시도하고, "여기서 멈추기"를 대안으로 저장
- **비탐욕적**: "여기서 멈추기"를 먼저 시도하고, "한 번 더 매칭하기"를 대안으로 저장

비유하자면, 뷔페에서 음식을 담는 두 가지 전략입니다:
- **탐욕적**: 접시에 최대한 많이 담은 다음, 너무 많으면 하나씩 빼기
- **비탐욕적**: 최소한만 담은 다음, 부족하면 하나씩 추가하기

### 핵심 개념 설명

#### 탐욕적 수량자의 내부 동작

`a*`(탐욕적)를 만났을 때 NFA 엔진의 결정 과정:

```
현재 위치에서 'a'를 읽을 수 있는가?
├── YES → "한 번 더 매칭" (우선 시도)
│         └── 백트래킹 스택에 "여기서 멈추기" 대안 저장
└── NO  → "여기서 멈추기" (더 이상 매칭 불가)
```

이를 반복하면, 탐욕적 수량자는 **가능한 모든 문자를 먼저 삼킨 후**, 뒤의 패턴이 실패하면 하나씩 토해내는 동작이 됩니다.

#### 비탐욕적 수량자의 내부 동작

`a*?`(비탐욕적)를 만났을 때:

```
현재 위치에서 'a'를 읽을 수 있는가?
├── YES → "여기서 멈추기" (우선 시도)
│         └── 백트래킹 스택에 "한 번 더 매칭" 대안 저장
└── NO  → "여기서 멈추기" (더 이상 매칭 불가)
```

비탐욕적 수량자는 **매칭하지 않는 것을 우선 시도**하고, 뒤의 패턴이 실패하면 하나씩 더 삼키는 동작이 됩니다.

#### 핵심 통찰: 결과가 같을 때도 과정이 다르다

중요한 사실이 있습니다: 탐욕적과 비탐욕적이 **같은 결과**를 내더라도, 엔진 내부의 **경로(과정)**는 다릅니다. 이는 성능 차이로 이어질 수 있습니다.

```python
import re

# 두 패턴 모두 같은 결과를 반환합니다
text = "aaa"

re.search(r'a+', text).group()    # 'aaa' (탐욕적)
re.search(r'a+?', text).group()   # 'a'   (비탐욕적) — 결과가 다름!

# 하지만 fullmatch에서는?
re.fullmatch(r'a+', text).group()    # 'aaa'
re.fullmatch(r'a+?', text).group()   # 'aaa' — 결과가 같음!
```

`fullmatch`에서 둘 다 `'aaa'`를 반환하지만, 내부 동작은 다릅니다:

**탐욕적 `a+`의 `fullmatch` 동작**:
```
a+가 'aaa' 전체를 삼킴 (3글자)
패턴 끝 & 문자열 끝 → 즉시 성공! (백트래킹 0회)
```

**비탐욕적 `a+?`의 `fullmatch` 동작**:
```
a+?가 'a' 1글자만 삼김
패턴 끝이지만 문자열이 안 끝남 ❌
🔙 백트래킹: a+?가 1글자 더 삼김 → 'aa' (2글자)
패턴 끝이지만 문자열이 안 끝남 ❌
🔙 백트래킹: a+?가 1글자 더 삼김 → 'aaa' (3글자)
패턴 끝 & 문자열 끝 → 성공! (백트래킹 2회)
```

결과는 같지만, 비탐욕적 버전은 백트래킹이 2회 추가로 발생했습니다. 이처럼 **"어차피 전체를 매칭해야 하는 상황"에서는 탐욕적 수량자가 더 효율적**입니다.

### 상세 예시

> 🔍 **예시 15.4.1: 탐욕적 vs 비탐욕적의 엔진 동작 비교**
>
> **상황**: HTML 태그에서 첫 번째 태그만 추출하는 두 가지 접근
>
> ```python
> import re
> text = '<b>bold</b> and <i>italic</i>'
> ```
>
> **풀이/설명**:
>
> **패턴 1: `<.*>` (탐욕적)**
> ```
> '<' 매칭 ✅ (위치 0)
> .* 탐욕적으로 삼킴: 'b>bold</b> and <i>italic</i' (위치 1~28)
>   → 문자열 끝까지 전부 삼킴
> '>' 매칭: 문자열 끝 ❌
> 🔙 .* 1글자 토해냄 → 현재 위치 28
> '>' 매칭: 위치 28 '>' ❌ (실제로는 '>')
>   → 아, 마지막 문자가 '>'이므로 매칭 성공!
> 
> 결과: '<b>bold</b> and <i>italic</i>' (전체!)
> 백트래킹: 1회
> ```
>
> **패턴 2: `<.*?>` (비탐욕적)**
> ```
> '<' 매칭 ✅ (위치 0)
> .*? 비탐욕적으로 0글자 삼킴
> '>' 매칭: 위치 1 'b' ❌
> 🔙 .*? 1글자 더 삼킴 → 'b' (1글자)
> '>' 매칭: 위치 2 '>' ✅
> 
> 결과: '<b>' (첫 번째 태그만!)
> 백트래킹: 1회
> ```
>
> **패턴 3: `<[^>]*>` (부정 문자 클래스)**
> ```
> '<' 매칭 ✅ (위치 0)
> [^>]* 삼킴: 'b' (1글자, '>'가 아닌 것만)
>   → 위치 2에서 '>' 만남 → [^>]* 멈춤
> '>' 매칭: 위치 2 '>' ✅
> 
> 결과: '<b>' (첫 번째 태그만!)
> 백트래킹: 0회
> ```
>
> **핵심 포인트**: 세 패턴의 결과와 백트래킹 횟수가 모두 다릅니다. 패턴 3(`<[^>]*>`)이 가장 효율적입니다. 비탐욕적 수량자(`.*?`)는 탐욕적보다 원하는 결과를 더 쉽게 얻을 수 있지만, 부정 문자 클래스가 가능하다면 그것이 가장 좋은 선택입니다.

> 🔍 **예시 15.4.2: `?` 수량자의 탐욕적/비탐욕적 동작**
>
> **상황**: 선택적 요소를 포함하는 패턴에서의 차이
>
> ```python
> import re
> 
> text = "colour"
> 
> # 탐욕적 ?: 'u'를 매칭하는 것을 먼저 시도
> re.search(r'colou?r', text).group()    # 'colour'
> 
> # 비탐욕적 ??: 'u'를 매칭하지 않는 것을 먼저 시도
> re.search(r'colou??r', text).group()   # 'colour'
> ```
>
> **풀이/설명**:
>
> **탐욕적 `u?`**:
> ```
> 'c','o','l','o' 매칭 ✅
> u? → 'u'를 매칭하는 것을 먼저 시도 → 'u' ✅
> 'r' 매칭 → 'r' ✅ → 성공! (백트래킹 0회)
> ```
>
> **비탐욕적 `u??`**:
> ```
> 'c','o','l','o' 매칭 ✅
> u?? → 'u'를 매칭하지 않는 것을 먼저 시도
> 'r' 매칭 → 위치의 문자 'u' ≠ 'r' ❌
> 🔙 백트래킹: u??가 'u'를 매칭
> 'r' 매칭 → 'r' ✅ → 성공! (백트래킹 1회)
> ```
>
> 결과는 같지만 비탐욕적 버전에서 불필요한 백트래킹이 1회 발생했습니다.
>
> **핵심 포인트**: `?`와 `??`는 결과가 같은 경우가 많지만, `??`는 "매칭하지 않기"를 먼저 시도하므로 뒤의 패턴과 충돌할 때 추가 백트래킹이 발생할 수 있습니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: "비탐욕적 수량자가 항상 더 빠르다"는 흔한 오해입니다. 비탐욕적 수량자는 **짧은 매칭을 먼저 시도**할 뿐이며, 긴 매칭이 필요한 상황에서는 오히려 탐욕적보다 더 많은 백트래킹이 발생합니다. 성능 최적화의 핵심은 탐욕적/비탐욕적 선택이 아니라, **불필요한 백트래킹 자체를 줄이는 것**입니다(구체적 문자 클래스 사용, 앵커 활용 등).

> ⚠️ **주의**: `.*`와 `.*?`는 동작 방향이 반대이므로 **결과도 다를 수 있습니다**. 단순히 "느려서" 비탐욕적으로 바꾸는 것이 아니라, 원하는 결과가 무엇인지 먼저 생각하고 적절한 수량자를 선택해야 합니다.

### Section 요약

- 탐욕적과 비탐욕적의 차이는 **분기점에서의 시도 순서**입니다: 탐욕적은 "더 매칭"을 먼저, 비탐욕적은 "멈추기"를 먼저 시도합니다
- 둘 다 **백트래킹을 통해 대안을 시도**하므로, 최종 결과가 같을 수 있지만 과정(백트래킹 횟수)은 다릅니다
- **부정 문자 클래스**(`[^X]*`)는 두 수량자 모두보다 효율적인 경우가 많은데, 백트래킹 자체를 줄이기 때문입니다
- 비탐욕적이 항상 빠른 것은 아닙니다. **상황에 따라 적절한 전략을 선택**해야 합니다

## 15.5 소유적 수량자와 원자적 그룹

> 💡 **한 줄 요약**: 소유적 수량자와 원자적 그룹은 "한 번 삼킨 문자를 절대로 토해내지 않는" 수량자/그룹으로, 백트래킹을 원천 차단하여 성능을 획기적으로 개선합니다.

### 직관적 이해

탐욕적 수량자가 뷔페에서 "최대한 담고, 필요하면 뺀다"는 전략이었다면, **소유적 수량자(possessive quantifier)**는 "최대한 담고, **절대로 빼지 않는다**"는 전략입니다.

한 번 삼킨 문자는 어떤 일이 있어도 토해내지 않습니다. 뒤의 패턴이 매칭에 실패해도, 이미 삼킨 부분으로 되돌아가지 않습니다. 성공하면 성공, 실패하면 그냥 실패입니다.

왜 이런 "고집스러운" 수량자가 필요할까요? 바로 **재앙적 백트래킹을 방지**하기 위해서입니다. 앞서 15.3에서 `(a+)+b` 패턴이 지수적 폭발을 일으키는 것을 보았습니다. 만약 안쪽 `a+`가 소유적이라면, 한 번 삼킨 `'a'`들을 절대 토해내지 않으므로 폭발이 발생하지 않습니다.

### 핵심 개념 설명

#### 소유적 수량자

> 📝 **소유적 수량자(Possessive Quantifier)**: 탐욕적 수량자처럼 가능한 한 많이 매칭하되, **백트래킹을 허용하지 않는** 수량자입니다. 한 번 매칭한 문자를 절대로 포기하지 않습니다.

소유적 수량자의 표기법은 탐욕적 수량자 뒤에 `+`를 붙입니다:

| 탐욕적 | 비탐욕적 | 소유적 |
|--------|---------|--------|
| `*` | `*?` | `*+` |
| `+` | `+?` | `++` |
| `?` | `??` | `?+` |
| `{n,m}` | `{n,m}?` | `{n,m}+` |

세 가지의 동작 차이를 비교합니다:

```
패턴: a___b  (___에 수량자 종류별 대입)
입력: "aaab"

탐욕적 (a*b):
  a* 가 "aaa" 삼킴 → b 매칭 시도 'b' ✅ → 성공
  (필요하면 백트래킹 가능)

비탐욕적 (a*?b):
  a*? 가 "" 삼킴 → b 매칭 시도 'a' ❌
  a*? 가 "a" 삼킴 → b 매칭 시도 'a' ❌
  a*? 가 "aa" 삼킴 → b 매칭 시도 'a' ❌
  a*? 가 "aaa" 삼킴 → b 매칭 시도 'b' ✅ → 성공

소유적 (a*+b):
  a*+ 가 "aaa" 삼킴 → b 매칭 시도 'b' ✅ → 성공
  (백트래킹 불가, 하지만 이 경우 필요 없었음)
```

이 예시에서는 세 가지 모두 성공합니다. 차이가 드러나는 경우를 봅시다:

```
패턴: a___a  (___에 수량자 종류별 대입)
입력: "aaaa"

탐욕적 (a*a):
  a* 가 "aaaa" 삼킴 → a 매칭 시도: 문자열 끝 ❌
  🔙 a* 가 "aaa"로 줄임 → a 매칭 시도: 'a' ✅ → 성공! ("aaaa")

소유적 (a*+a):
  a*+ 가 "aaaa" 삼킴 → a 매칭 시도: 문자열 끝 ❌
  ❌ 백트래킹 불가 → 매칭 실패!
```

소유적 `a*+`는 모든 `'a'`를 삼킨 후 뒤의 `a`에게 내줄 수 있는 문자가 없고, 백트래킹도 하지 않으므로 실패합니다.

> ⚠️ **주의**: 소유적 수량자는 **파이썬 표준 `re` 모듈에서는 지원하지 않습니다**. PyPI의 `regex` 모듈에서 사용할 수 있습니다. 여기서는 개념과 원리를 이해하는 데 초점을 둡니다. `regex` 모듈의 실제 사용법은 Chapter 17에서 다룹니다.

#### 원자적 그룹

> 📝 **원자적 그룹(Atomic Group)**: `(?>...)`으로 표기하며, 그룹 내부의 매칭이 완료된 후에는 그룹 내부로 되돌아가는 **백트래킹을 허용하지 않는** 그룹입니다. 그룹이 성공적으로 매칭되면, 그 결과는 "원자적으로(더 이상 쪼갤 수 없게)" 확정됩니다.

```
일반 그룹:   (a*)     → 백트래킹 시 내부로 돌아갈 수 있음
원자적 그룹: (?>a*)   → 백트래킹 시 내부로 돌아갈 수 없음
```

원자적 그룹은 소유적 수량자의 **일반화된 형태**입니다:
- `a*+`는 `(?>a*)`와 동일하게 동작합니다
- 하지만 원자적 그룹은 수량자 외에도 선택(`|`)이나 복잡한 패턴을 감쌀 수 있어 더 유연합니다

```
소유적 수량자:  a*+        (수량자에만 적용 가능)
원자적 그룹:   (?>a*)      (수량자에 적용)
원자적 그룹:   (?>ab|cd)   (선택에도 적용 가능!)
```

선택에 원자적 그룹을 적용하면:

```
패턴: (?>cat|car)s
입력: "cars"

(?>cat|car):
  선택지 1 'cat' 시도: 'c' ✅, 'a' ✅, 't' != 'r' ❌
  선택지 2 'car' 시도: 'c' ✅, 'a' ✅, 'r' ✅ → 그룹 매칭 확정!
's' 매칭: 's' ✅ → 성공!

패턴: (?>car|cat)s
입력: "cats"

(?>car|cat):
  선택지 1 'car' 시도: 'c' ✅, 'a' ✅, 'r' != 't' ❌
  선택지 2 'cat' 시도: 'c' ✅, 'a' ✅, 't' ✅ → 그룹 매칭 확정! ("cat")
's' 매칭: 's' ✅ → 성공!
```

원자적 그룹 내부에서는 정상적으로 백트래킹이 발생합니다(선택지 1 실패 → 선택지 2 시도). 하지만 그룹이 확정된 **이후**에 외부에서 그룹 내부로 되돌아가는 백트래킹은 차단됩니다.

#### 소유적 수량자/원자적 그룹의 실전 가치

이들의 가치는 **매칭이 실패할 때** 극적으로 드러납니다:

```
패턴: \d+:
입력: "abcdefghij" (숫자가 전혀 없는 긴 문자열)

일반 탐욕적 \d+:
  위치 0: \d+ → 'a'는 숫자 아님 → \d+ 자체가 실패 (1회 이상 필요)
  → 위치 이동... (각 위치에서 같은 실패 반복)

이 경우는 차이가 없습니다.
```

차이가 나는 진짜 시나리오:

```
패턴: \d+\d+failure
입력: "12345678901234567890" (20자리 숫자, 뒤에 'failure' 없음)

일반 탐욕적:
  첫 \d+가 20자리 삼킴 → 두 번째 \d+에게 남은 것 없음
  🔙 백트래킹: 첫 \d+가 19자리, 두 번째가 1자리 → 'failure' 없음 ❌
  🔙 백트래킹: 첫 \d+가 18자리, 두 번째가 2자리 → 'failure' 없음 ❌
  ... (수많은 조합을 시도)
  → 최종 실패 (하지만 엄청난 백트래킹 발생)

소유적 수량자 \d++\d+failure:
  첫 \d++가 20자리 삼킴 (소유적 → 절대 토해내지 않음)
  두 번째 \d+에게 남은 것 없음 → 즉시 실패!
  → 불필요한 백트래킹 없이 빠르게 실패 확정
```

> 💬 **참고**: 소유적 수량자와 원자적 그룹의 실제 코드 사용법은 Chapter 17에서 `regex` 모듈과 함께 다룹니다. 이번 Section에서는 원리를 이해하는 것에 집중합니다.

### 상세 예시

> 🔍 **예시 15.5.1: 소유적 수량자의 동작 원리**
>
> **상황**: 패턴 `".*+"` (소유적)와 `".*"` (탐욕적)의 동작 차이
>
> **풀이/설명**:
>
> 입력: `'She said "hello"'`
>
> **탐욕적 `".*"`**:
> ```
> '"' 매칭 ✅ (위치 9)
> .* 가 'hello"' 전부 삼킴 (위치 10~15)
> '"' 매칭 시도: 문자열 끝 ❌
> 🔙 .* 가 '"' 토해냄 (위치 15)
> '"' 매칭: '"' ✅ → 성공! 결과: '"hello"'
> ```
>
> **소유적 `".*+"`** (개념적 동작):
> ```
> '"' 매칭 ✅ (위치 9)
> .*+ 가 'hello"' 전부 삼킴 (위치 10~15)
> '"' 매칭 시도: 문자열 끝 ❌
> ❌ 소유적이므로 백트래킹 불가 → 매칭 실패!
> ```
>
> **핵심 포인트**: 소유적 수량자는 유용하지만 사용에 주의가 필요합니다. `".*+"`는 닫는 따옴표를 찾을 수 없어 실패합니다. 소유적 수량자는 "이 부분이 토해낼 필요가 절대 없다고 확신할 때"만 사용해야 합니다. 이 경우 `"[^"]*+"`처럼 부정 문자 클래스와 결합하면 안전합니다.

> 🔍 **예시 15.5.2: 원자적 그룹으로 재앙적 백트래킹 방지**
>
> **상황**: Section 15.3에서 본 `(a+)+b` 패턴의 재앙적 백트래킹 문제를 원자적 그룹으로 해결
>
> **풀이/설명**:
>
> 문제 패턴: `(a+)+b` — 입력 `"aaaa"`에서 지수적 백트래킹 발생
>
> 해결 패턴: `(?>a+)+b` — 원자적 그룹 적용
>
> ```
> 입력: "aaaa" (끝에 'b' 없음)
>
> (?>a+)+b 의 동작:
>   (?>a+) 첫 번째 시도: a+가 "aaaa" 전부 삼킴
>     → 원자적 그룹 확정! "aaaa"는 되돌릴 수 없음
>   다시 (?>a+) 시도: 남은 문자 없음 → 반복 종료
>   'b' 매칭: 문자열 끝 ❌
>   🔙 외부 +가 백트래킹하려 하지만, (?>a+)는 원자적이므로
>      내부로 돌아가 "aaa" + "a"로 재분배할 수 없음
>   → 즉시 매칭 실패! (백트래킹 최소화)
> ```
>
> 원자적 그룹이 없었다면 `(a+)+b`는 "aaaa"를 (aaaa), (aaa)(a), (aa)(aa), ... 등 2^4 = 16가지 이상의 조합을 시도했을 것입니다.
>
> **핵심 포인트**: 원자적 그룹은 내부 매칭 결과를 "봉인"하여, 외부 백트래킹이 내부를 재분배하지 못하게 합니다. 이것이 재앙적 백트래킹의 근본적 해결책입니다.

### 깊이 있는 이해 (Deep Dive)

> 🔬 **Deep Dive: 파이썬 re 모듈에서의 원자적 그룹 대안**
>
> *이 부분은 심화 내용입니다. 처음 학습 시 건너뛰어도 됩니다.*
>
> 파이썬의 표준 `re` 모듈은 소유적 수량자(`*+`)와 원자적 그룹(`(?>...)`)을 지원하지 않습니다. 하지만 일부 상황에서 비슷한 효과를 달성할 수 있는 우회 방법이 있습니다.
>
> **방법: 전방 탐색 + 역참조를 이용한 원자적 그룹 흉내**
>
> > 🔗 **연결**: 이 기법은 Chapter 9의 전방 탐색 `(?=...)`과 Chapter 8의 역참조를 결합합니다.
>
> ```python
> import re
> 
> # 원자적 그룹: (?>pattern)
> # re 모듈 대안: (?=(pattern))\1
> 
> # 예: (?>a+) 의 대안
> # (?=(a+))\1
> ```
>
> 이 트릭의 원리:
> 1. `(?=(a+))` — 전방 탐색 안에서 `a+`가 탐욕적으로 매칭 (하지만 소비하지 않음)
> 2. `\1` — 전방 탐색에서 캡처한 내용을 역참조로 실제 소비
>
> 전방 탐색은 제로 너비 단언이므로 내부 매칭이 완료된 후에는 백트래킹할 수 없습니다. 따라서 원자적 그룹과 유사한 효과를 냅니다.
>
> 다만 이 방법은 패턴이 복잡해지고 가독성이 떨어지므로, **`regex` 모듈을 사용할 수 있다면 그것이 더 나은 선택**입니다. `regex` 모듈의 사용법은 Chapter 17에서 다룹니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: 소유적 수량자를 무분별하게 사용하면 **정상적인 매칭까지 실패**할 수 있습니다. 소유적 수량자는 "이 수량자가 삼킨 문자를 뒤의 패턴에 양보할 필요가 없다"고 확신할 때만 사용해야 합니다. 확신이 없다면 일반 탐욕적/비탐욕적 수량자를 사용하는 것이 안전합니다.

> ⚠️ **주의**: 소유적 수량자(`*+`, `++`)와 원자적 그룹(`(?>...)`)은 **파이썬 표준 `re` 모듈에서 지원하지 않습니다**. 이 구문을 `re` 모듈에서 사용하면 에러가 발생합니다. 사용하려면 PyPI의 `regex` 모듈을 설치해야 합니다.

### Section 요약

- **소유적 수량자**(`*+`, `++`, `?+`)는 탐욕적 수량자처럼 최대한 삼키되, **절대로 토해내지 않습니다** (백트래킹 불가)
- **원자적 그룹**(`(?>...)`)은 그룹 내부 매칭을 확정한 후, 외부에서 내부로의 백트래킹을 차단합니다
- 소유적 수량자는 원자적 그룹의 특수한 경우이며, 원자적 그룹이 더 유연합니다
- 이들의 핵심 가치는 **재앙적 백트래킹 방지**에 있습니다
- 파이썬 표준 `re` 모듈에서는 지원하지 않으며, `regex` 모듈을 통해 사용할 수 있습니다

## 15.6 DFA vs NFA

> 💡 **한 줄 요약**: DFA 엔진은 입력 길이에 비례하는 예측 가능한 성능을 제공하지만, NFA 엔진만이 캡처 그룹, 역참조, 전후방 탐색 같은 고급 기능을 지원합니다.

### 직관적 이해

Section 15.1에서 미로 탈출의 두 전략을 비유했습니다. 이제 이 두 전략을 본격적으로 비교합니다.

**NFA (전략 A: 한 길씩 시도)**
- 한 명의 탐험가가 갈림길마다 메모를 남기고, 막다른 길이면 돌아옴
- 장점: 탐험가가 정확히 **어느 경로로 왔는지** 기억함 (= 캡처 그룹)
- 단점: 운이 나쁘면 같은 장소를 여러 번 방문할 수 있음 (= 백트래킹 비용)

**DFA (전략 B: 동시 탐색)**
- 분신술로 갈림길마다 복제를 보내, 모든 경로를 **동시에** 탐색
- 장점: 미로의 모든 길을 정확히 한 번씩만 확인 (= 성능 보장)
- 단점: 어느 분신이 어떤 경로로 왔는지 구분 불가 (= 캡처 불가)

### 핵심 개념 설명

#### DFA의 동작 방식

DFA 엔진은 NFA와 근본적으로 다르게 동작합니다:

**DFA는 입력 주도(Text-driven)**입니다. 입력 문자를 하나 읽을 때마다, 현재 가능한 **모든 상태를 동시에** 추적합니다.

패턴 `cat|car`에 대해 `"car"`를 매칭하는 과정:

```
DFA의 동작:

입력 'c' 읽음:
  → 가능한 상태: {cat의 'c' 이후, car의 'c' 이후}
  → 두 상태를 동시에 유지

입력 'a' 읽음:
  → 가능한 상태: {cat의 'ca' 이후, car의 'ca' 이후}
  → 여전히 두 상태 동시 유지

입력 'r' 읽음:
  → cat 경로: 'r' ≠ 't' → 탈락
  → car 경로: 'r' == 'r' → 수락 상태 도달!
  → 매칭 성공: "car"
```

NFA와의 핵심 차이에 주목하세요:
- NFA: `cat`을 먼저 시도 → 실패 → 백트래킹 → `car` 시도 → 성공 (순차적)
- DFA: `cat`과 `car`을 **동시에** 추적 → `cat` 탈락, `car` 수락 (병렬적)

DFA는 **입력의 각 문자를 정확히 한 번만 읽습니다**. 백트래킹이 없으므로, 매칭 시간은 **항상 입력 문자열의 길이에 비례**합니다.

#### 두 엔진의 상세 비교

| 특성 | NFA | DFA |
|------|-----|-----|
| **동작 방식** | 패턴 주도 (한 경로씩) | 입력 주도 (모든 경로 동시) |
| **백트래킹** | 있음 | 없음 |
| **시간 복잡도** | 최악 O(2^n), 보통 O(n) | 항상 O(n) |
| **메모리** | 백트래킹 스택 (가변) | 상태 집합 (고정적) |
| **캡처 그룹** | ✅ 지원 | ❌ 지원 불가 |
| **역참조** | ✅ 지원 | ❌ 지원 불가 |
| **전후방 탐색** | ✅ 지원 | ❌ 지원 불가 |
| **소유적 수량자** | ✅ 개념 존재 | 해당 없음 |
| **탐욕적/비탐욕적 구분** | ✅ 결과에 영향 | ❌ 구분 없음 (항상 최장 매칭) |
| **사용 예시** | Python, Java, Perl, JS, Ruby | grep (전통), awk, lex |

#### DFA가 캡처 그룹을 지원할 수 없는 이유

DFA는 "모든 경로를 동시에 추적"하기 때문에, **어떤 특정 경로를 통해 매칭에 성공했는지** 알 수 없습니다. 캡처 그룹은 "이 부분 괄호가 어떤 텍스트와 매칭되었는지"를 기록해야 하는데, 경로 정보가 없으면 이것이 불가능합니다.

```python
# NFA라서 가능한 것들:
import re

# 1. 캡처 그룹
m = re.search(r'(\d+)-(\d+)', '2024-01-15')
m.group(1)  # '2024' — 첫 번째 괄호가 캡처한 내용

# 2. 역참조 (Chapter 8)
re.search(r'(\w+) \1', 'hello hello')  # 같은 단어 반복 찾기

# 3. 전후방 탐색 (Chapter 9)
re.findall(r'(?<=\$)\d+', 'Price: $100')  # '$' 뒤의 숫자
```

이 세 가지 기능 모두 "특정 경로의 매칭 상태"를 추적해야 하므로, DFA로는 구현할 수 없습니다.

#### 현실의 하이브리드 엔진

실제 소프트웨어에서는 두 가지를 혼합하기도 합니다:

- **GNU grep**: 기본적으로 DFA를 사용하되, 역참조(`\1`)가 포함된 패턴에서는 NFA로 전환
- **RE2** (Google): DFA 기반이며, NFA 전용 기능(역참조 등)을 아예 지원하지 않음. 대신 어떤 입력이 들어와도 성능이 보장됨
- **파이썬 `re`**: 순수 NFA
- **`regex` 모듈** (PyPI): NFA 기반이지만, 다양한 최적화 적용

### 상세 예시

> 🔍 **예시 15.6.1: NFA와 DFA의 매칭 결과 차이**
>
> **상황**: 패턴 `to(nite|knight|night)`에 대해 `"tonight"` 매칭
>
> **풀이/설명**:
>
> **NFA 엔진**:
> ```
> 'to' 매칭 ✅
> 선택지 1 'nite' 시도:
>   'n' ✅, 'i' ✅, 't' != 'g' ❌
> 🔙 백트래킹
> 선택지 2 'knight' 시도:
>   'k' != 'n' ❌
> 🔙 백트래킹
> 선택지 3 'night' 시도:
>   'n' ✅, 'i' ✅, 'g' ✅, 'h' ✅, 't' ✅ → 성공!
> 결과: "tonight"
> ```
>
> **DFA 엔진**:
> ```
> 't' 읽음 → {모든 선택지 추적 시작}
> 'o' 읽음 → {nite, knight, night 모두 진행}
> 'n' 읽음 → {nite: 'n'✅, knight: 'k'≠'n' 탈락, night: 'n'✅}
>          → {nite, night 2개 남음}
> 'i' 읽음 → {nite: 'i'✅, night: 'i'✅}
> 'g' 읽음 → {nite: 't'≠'g' 탈락, night: 'g'✅}
>          → {night 1개 남음}
> 'h' 읽음 → {night: 'h'✅}
> 't' 읽음 → {night: 't'✅ → 수락!}
> 결과: "tonight"
> ```
>
> **핵심 포인트**: 두 엔진의 결과는 같지만, 과정이 다릅니다. NFA는 순차적으로 시도하며 백트래킹이 발생합니다. DFA는 모든 가능성을 동시에 추적하며 각 문자를 정확히 한 번만 읽습니다. 이 예시에서 NFA는 `'nite'`의 3글자와 `'knight'`의 1글자를 "낭비"했지만, DFA는 7글자를 딱 한 번씩만 읽었습니다.

> 🔍 **예시 15.6.2: 성능 특성의 극적인 차이**
>
> **상황**: 패턴 `a?a?a?aaa`에 입력 `"aaa"` (간략화된 예시)
>
> 이 패턴은 유명한 벤치마크 예시의 축소판입니다. `a?`가 n개, `a`가 n개인 패턴에 `"a" * n` 문자열을 매칭할 때:
>
> **풀이/설명**:
>
> **DFA**: `a?`가 n개이든 100개이든, 입력 문자열의 길이 n에 비례하여 항상 O(n)으로 완료
>
> **NFA**: `a?`가 탐욕적으로 `'a'`를 삼킨 후, 뒤의 `a`에 양보하기 위한 백트래킹이 발생. n이 커지면 O(2^n)에 근접할 수 있음
>
> | 입력 길이 (n) | NFA 시도 횟수 (대략) | DFA 시도 횟수 |
> |-------------|---------------------|-------------|
> | 10 | ~1,000 | 10 |
> | 20 | ~1,000,000 | 20 |
> | 30 | ~1,000,000,000 | 30 |
>
> **핵심 포인트**: 이것이 DFA 기반 엔진(예: RE2)이 존재하는 이유입니다. 보안이 중요한 환경(웹 서버의 입력 검증 등)에서는 NFA의 예측 불가능한 성능이 위험(ReDoS 공격)이 될 수 있으며, DFA 기반 엔진이 더 안전할 수 있습니다. 이 주제는 Chapter 16에서 더 깊이 다룹니다.

### 흔한 실수와 주의사항

> ⚠️ **주의**: "DFA가 항상 더 좋다"거나 "NFA가 항상 더 좋다"고 단정짓지 마세요. 두 엔진은 서로 다른 **트레이드오프**를 가집니다. 캡처 그룹과 역참조가 필요하면 NFA를 써야 하고, 성능 보장이 최우선이면 DFA(또는 DFA 기반 도구)가 적합합니다. 대부분의 프로그래밍 작업에서는 NFA의 풍부한 기능이 더 유용합니다.

> ⚠️ **주의**: NFA 엔진에서 선택(`|`)의 순서가 결과에 영향을 주지만, DFA에서는 **순서가 결과에 영향을 주지 않습니다** (항상 가장 긴 매칭 반환). 이 차이를 알면 다른 도구(예: `grep`)와 파이썬의 결과가 다를 때 원인을 이해할 수 있습니다.

### Section 요약

- **DFA**는 입력 주도로 모든 경로를 동시에 추적하며, **항상 O(n)** 시간에 매칭을 완료합니다
- **NFA**는 패턴 주도로 한 경로씩 시도하며, 백트래킹으로 인해 **최악 O(2^n)**이 될 수 있습니다
- DFA는 **캡처 그룹, 역참조, 전후방 탐색을 지원할 수 없어**, 대부분의 프로그래밍 언어가 NFA를 채택합니다
- 실제 소프트웨어에서는 하이브리드 엔진이나 상황별 전환을 사용하기도 합니다
- 보안이 중요한 환경에서는 DFA의 성능 보장이 큰 장점이 됩니다

## Chapter 핵심 요약 (Chapter Summary)

이 Chapter에서는 정규표현식의 "무대 뒤"를 들여다보았습니다. 우리가 `re.search()`를 호출할 때 파이썬 내부에서 실제로 벌어지는 일을 이해하게 되었습니다.

- **15.1 유한 오토마타와 정규표현식**: 정규표현식은 유한 오토마타라는 수학적 기계 모델에 기반합니다. NFA(여러 경로 가능)와 DFA(경로가 하나로 결정)의 두 종류가 있으며, 파이썬은 NFA 기반 엔진을 사용합니다.

- **15.2 NFA 기반 매칭의 동작**: NFA 엔진은 패턴 주도 방식으로, 왼쪽에서 오른쪽으로 패턴을 처리합니다. 분기점에서 상태를 저장하고 한 경로를 먼저 시도하며, 실패 시 저장점으로 백트래킹합니다. 선택(`|`)에서는 왼쪽 선택지가 우선입니다.

- **15.3 백트래킹 심화**: 백트래킹 스택은 LIFO 구조로 관리됩니다. 탐욕적 수량자는 "최대 삼킨 후 토해내며" 백트래킹하고, 비탐욕적은 "최소 삼킨 후 더 삼키며" 백트래킹합니다. 중첩 수량자는 지수적 폭발(재앙적 백트래킹)을 일으킬 수 있습니다.

- **15.4 탐욕적/비탐욕적 매칭의 엔진 수준 이해**: 두 매칭 전략의 차이는 분기점에서의 시도 순서에 불과합니다. 같은 결과를 내더라도 백트래킹 횟수가 다를 수 있으며, 부정 문자 클래스가 두 전략 모두보다 효율적인 경우가 많습니다.

- **15.5 소유적 수량자와 원자적 그룹**: 소유적 수량자(`*+`, `++`)와 원자적 그룹(`(?>...)`)은 백트래킹을 원천 차단하여 재앙적 백트래킹을 방지합니다. 파이썬 표준 `re` 모듈에서는 지원하지 않으며, `regex` 모듈에서 사용 가능합니다.

- **15.6 DFA vs NFA**: DFA는 성능이 보장되지만(항상 O(n)), 캡처 그룹과 역참조를 지원할 수 없습니다. NFA는 풍부한 기능을 제공하지만, 패턴에 따라 성능이 크게 달라질 수 있습니다.

## 핵심 용어 정리 (Glossary)

| 용어 | 정의 | 관련 Section |
|------|------|-------------|
| **유한 오토마타 (Finite Automaton)** | 유한한 상태들 사이를 입력에 따라 전이하며, 수락 상태에 도달하면 매칭 성공으로 판단하는 추상적 기계 모델 | 15.1 |
| **NFA (Nondeterministic Finite Automaton)** | 같은 입력에 대해 여러 전이가 가능하고, 엡실론 전이(문자 소비 없는 전이)를 허용하는 오토마타. 파이썬 `re` 모듈의 기반 | 15.1, 15.2 |
| **DFA (Deterministic Finite Automaton)** | 각 상태에서 특정 입력에 대한 전이가 하나로 결정되는 오토마타. 항상 O(n) 성능 보장 | 15.1, 15.6 |
| **엡실론 전이 (ε-transition)** | NFA에서 입력 문자를 소비하지 않고 수행하는 상태 전이 | 15.1 |
| **패턴 주도 (Pattern-driven)** | NFA의 동작 방식. 패턴의 각 요소를 기준으로 입력을 대조 | 15.2 |
| **입력 주도 (Text-driven)** | DFA의 동작 방식. 입력 문자를 기준으로 가능한 모든 상태를 추적 | 15.6 |
| **백트래킹 스택 (Backtracking Stack)** | NFA 엔진이 분기점에서 대안 경로의 상태를 저장하는 LIFO 자료구조 | 15.3 |
| **재앙적 백트래킹 (Catastrophic Backtracking)** | 중첩 수량자 등으로 인해 백트래킹 횟수가 지수적으로 폭발하는 현상 | 15.3 |
| **소유적 수량자 (Possessive Quantifier)** | `*+`, `++` 등. 탐욕적으로 매칭하되 백트래킹을 허용하지 않는 수량자 | 15.5 |
| **원자적 그룹 (Atomic Group)** | `(?>...)`. 내부 매칭 확정 후 외부에서 내부로의 백트래킹을 차단하는 그룹 | 15.5 |

</details>

>

> ```

>

> '<' 매칭 ✅ (위치 0)

> '<' 매칭 ✅ (위치 0)
> .*? → 0글자
> '>' 매칭: 위치 1 'd' ❌

> </details>

> **접근 방법**: 중첩 수량자의 재앙적 백트래킹 패턴을 분석합니다.
>
> **(a)** 최종 결과: `None` (매칭 실패). 문자열에 `'y'`가 없으므로 어떤 분배를 시도해도 성공할 수 없습니다.
>
> **(b)** 실행이 오래 걸리는 이유:
>
> 바깥쪽 `(...)+ `는 그룹을 여러 번 반복할 수 있고, 각 반복에서 안쪽 `x+x+`는 'x'들을 두 부분으로 나눕니다. 10개의 'x'에 대해:
>
> - 그룹 1회 반복: (x를 1+9, 2+8, 3+7, ..., 9+1로 분배) = 9가지
> - 그룹 2회 반복: 첫 반복에서 k개 사용, 나머지를 두 번째 반복에서 분배 → 조합 급증
> - 그룹 3회 이상: 더욱 폭발
>
> 총 시도 횟수는 대략 2^n에 근접하여, n=10에서도 약 1,000번 이상, n=20이면 약 100만 번, n=30이면 약 10억 번의 시도가 필요합니다.
>
> **(c)** 안전한 개선 방법:
>
> ```python
> # 원래 문제 패턴
> r'(x+x+)+y'    # 위험! 재앙적 백트래킹
>
> # 개선 1: 중첩 제거 — x가 2개 이상 반복된 후 y
> r'x{2,}y'      # 안전하고 동일한 의미
>
> # 개선 2: x가 최소 2개 이상이면 된다면
> r'xx+y'        # 안전
> ```
>
> 핵심은 `(x+x+)+`가 결국 "x가 2개 이상"이라는 의미이므로, 중첩 수량자 없이 `x{2,}`로 단순화할 수 있다는 것입니다.
>
> **확장 생각**: 이러한 중첩 수량자 패턴은 실전에서도 의도치 않게 만들어질 수 있습니다. 예: `(\s+|\t+)+`는 공백 문자에 대해 비슷한 폭발을 일으킬 수 있습니다. Chapter 16에서 이런 패턴을 체계적으로 식별하는 방법을 배웁니다.
> </details>

## D3. Problem (심화 문제) — 깊이 있는 사고와 창의적 문제 해결

> **Problem 15.1**: 다음 패턴에 대해 NFA 엔진의 **전체 백트래킹 과정**을 상태 테이블 형식으로 추적하세요.
>
> ```python
> import re
> re.search(r'(a|ab)(bc|c)', 'abc')
> ```
>
> 각 단계에서 (1) 현재 시도 중인 패턴 위치, (2) 입력 위치, (3) 백트래킹 스택의 상태를 명시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 두 개의 그룹이 각각 선택지를 가지므로, NFA는 왼쪽 우선 규칙에 따라 각 그룹의 첫 번째 선택지를 우선 시도하고, 실패 시 순차적으로 백트래킹합니다.
>
> **풀이 전략**: 상태를 (패턴 위치, 입력 위치, 스택)으로 표현합니다.
>
> **상세 풀이**:
>
> ```
> 입력: "abc"
>        012
> 패턴: (a|ab)(bc|c)
>
> === 단계 1 ===
> 그룹1: 선택지 'a' 시도 (왼쪽 우선)
> 스택에 저장: [그룹1에서 'ab' 시도할 수 있음, 입력위치=0]
> 입력 위치 0: 'a' == 'a' ✅
> 그룹1 결과: 'a' 매칭, 입력 위치 → 1
>
> === 단계 2 ===
> 그룹2: 선택지 'bc' 시도 (왼쪽 우선)
> 스택에 저장: [그룹2에서 'c' 시도할 수 있음, 입력위치=1]
> 입력 위치 1: 'b' == 'b' ✅
> 입력 위치 2: 'c' == 'c' ✅
> 그룹2 결과: 'bc' 매칭
>
> === 단계 3 ===
> 패턴 끝 도달 & 매칭 성공!
>
> 결과: 'abc'
> group(1) = 'a', group(2) = 'bc'
> ```
>
> 흥미로운 점은, 첫 번째 시도에서 바로 성공했다는 것입니다. 하지만 만약 그룹1이 `'a'`를 먼저 매칭하지 않고 `'ab'`를 먼저 매칭했다면?
>
> ```
> 대안 시나리오 (만약 그룹1이 'ab'를 먼저 선택했다면):
> 그룹1: 'ab' 매칭, 입력 위치 → 2
> 그룹2: 'bc' 시도 → 위치 2 'c' ≠ 'b' ❌
> 그룹2: 'c' 시도 → 위치 2 'c' == 'c' ✅ → 성공!
> 결과: group(1) = 'ab', group(2) = 'c'
> ```
>
> 두 경우 모두 전체 매칭은 `'abc'`이지만, **캡처 그룹의 내용이 다릅니다**. NFA의 왼쪽 우선 규칙이 캡처 결과까지 결정하는 것입니다.
>
> **확장 생각**: 이 예시는 NFA 엔진에서 **같은 전체 매칭이라도 내부 분배(캡처)가 달라질 수 있음**을 보여줍니다. 선택지의 순서가 전체 매칭뿐 아니라 캡처 그룹의 내용까지 영향을 미칩니다.
> </details>

> **Problem 15.2**: 다음 시나리오를 분석하세요.
>
> 웹 서버에서 사용자 입력을 검증하기 위해 다음 정규표현식을 사용하고 있습니다:
> ```python
> import re
> pattern = r'^([a-zA-Z0-9]+\.)+[a-zA-Z]{2,}$'
> # 의도: 도메인 이름 검증 (예: "example.com", "sub.domain.org")
> ```
>
> (a) 이 패턴이 정상적인 도메인 `"sub.domain.com"`에 대해 어떻게 동작하는지 설명하세요.
> (b) 악의적 사용자가 `"aaaaaaaaaaaaaaaaaaaaaaaaaaaa!"`(a 28개 + !)를 입력했을 때 어떤 일이 벌어지는지 설명하세요.
> (c) 이 패턴의 보안 위험과 개선 방안을 제시하세요.
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 중첩 수량자 `([a-zA-Z0-9]+\.)+`가 마침표(`.`)가 없는 입력에 대해 재앙적 백트래킹을 일으킵니다.
>
> **풀이 전략**: 정상 입력과 악의적 입력에서의 엔진 동작을 비교합니다.
>
> **(a) 정상 동작** (`"sub.domain.com"`):
> ```
> ^ 매칭 ✅
> ([a-zA-Z0-9]+\.)+ :
>   1회: [a-zA-Z0-9]+ → "sub", \. → "." ✅
>   2회: [a-zA-Z0-9]+ → "domain", \. → "." ✅
>   3회 시도: [a-zA-Z0-9]+ → "com", \. → 문자열 끝 ❌ → 3회 실패
>   그룹 반복 종료 (2회 성공)
> [a-zA-Z]{2,} → "com" ✅
> $ → 문자열 끝 ✅
> 매칭 성공!
> ```
>
> **(b) 악의적 입력** (`"a" * 28 + "!"`):
>
> 문자열에 마침표(`.`)가 없으므로, `[a-zA-Z0-9]+`가 모든 'a'를 삼킨 후 `\.`에서 실패합니다. 그러면 엔진은 `[a-zA-Z0-9]+`가 삼킨 양을 줄여가며 재시도합니다.
>
> 바깥쪽 `(...)+`도 여러 번 반복하므로, 28개의 'a'를 어떻게 분배할지에 대한 조합이 폭발합니다:
> - (28개) + 실패
> - (27개)(1개) + 실패
> - (26개)(2개) + 실패, (26개)(1개)(1개) + 실패
> - ... (지수적으로 증가)
>
> 이론적으로 약 **2^28 ≈ 2.7억** 번의 시도가 필요하며, 서버가 수 초에서 수 분간 멈출 수 있습니다.
>
> **(c) 보안 위험과 개선**:
>
> 이것은 **ReDoS(Regular Expression Denial of Service)** 공격의 전형적인 패턴입니다. 악의적 사용자가 의도적으로 이런 입력을 보내 서버를 마비시킬 수 있습니다.
>
> 개선 방안:
> ```python
> # 개선 1: 중첩 수량자 제거
> r'^[a-zA-Z0-9]+(\.[a-zA-Z0-9]+)*\.[a-zA-Z]{2,}$'
> # 마침표를 그룹 밖에서 명시하여 중첩 제거
>
> # 개선 2: 입력 길이 제한 (정규표현식 외부)
> if len(user_input) > 253:  # 도메인 최대 길이
>     return False
>
> # 개선 3: 원자적 그룹 사용 (regex 모듈)
> # r'^(?>[a-zA-Z0-9]+\.)+[a-zA-Z]{2,}$'
> ```
>
> **확장 생각**: 이 문제는 Chapter 16에서 다룰 ReDoS의 핵심 사례입니다. 정규표현식의 엔진 원리를 이해하는 것이 보안에도 직결된다는 것을 보여줍니다.
> </details>

> **Problem 15.3**: NFA와 DFA 엔진의 특성을 종합적으로 비교하는 사고 문제입니다.
>
> 당신은 웹 방화벽(WAF) 시스템을 설계하고 있습니다. 이 시스템은 HTTP 요청의 URL, 헤더, 본문에서 악의적 패턴(SQL 인젝션, XSS 등)을 정규표현식으로 탐지합니다. 초당 10,000건의 요청을 처리해야 합니다.
>
> (a) NFA 엔진과 DFA 엔진 중 어느 것이 이 용도에 더 적합할까요? 그 이유는?
> (b) 각 엔진을 선택했을 때의 장점과 단점은?
> (c) 이 두 가지의 단점을 완화할 수 있는 현실적인 전략은?
>
> <details>
> <summary>풀이 가이드</summary>
>
> **핵심 아이디어**: 성능 보장과 기능 풍부함 사이의 트레이드오프를 실제 시스템 설계에 적용합니다.
>
> **(a)** **DFA 엔진이 더 적합합니다.**
>
> 이유:
> - 초당 10,000건 처리에는 **예측 가능한 성능**이 필수입니다
> - NFA는 악의적으로 조작된 입력에 의해 재앙적 백트래킹이 발생할 수 있으며, 이는 서비스 거부(DoS)로 이어집니다
> - WAF의 패턴은 주로 문자열/키워드 매칭 위주이므로, 캡처 그룹이나 역참조가 필수적이지 않은 경우가 많습니다
> - DFA는 입력 길이에 비례하는 O(n) 성능이 항상 보장됩니다
>
> **(b)** 장단점 비교:
>
> | | DFA 선택 시 | NFA 선택 시 |
> |---|---|---|
> | **장점** | 성능 보장 (ReDoS 면역), 일정한 처리 시간 | 복잡한 패턴 작성 가능 (역참조, 전후방 탐색) |
> | **단점** | 캡처 불가, 역참조 불가로 패턴 표현력 제한, 메모리 사용량이 클 수 있음 | 악의적 입력에 취약 (ReDoS), 성능 예측 불가 |
>
> **(c)** 현실적 완화 전략:
>
> 1. **DFA 기반 엔진 + 기능 보완**: Google의 RE2처럼 DFA 기반 엔진을 사용하되, 필요한 경우 애플리케이션 코드로 캡처 기능을 보완
> 2. **NFA 사용 시 안전장치**: 매칭 시간 제한(타임아웃 설정), 입력 길이 제한, 패턴 사전 검증(ReDoS 취약 패턴 감지)
> 3. **하이브리드 접근**: 1차 필터링은 DFA(빠른 거부), 2차 상세 분석은 NFA(기능 활용)
> 4. **패턴 작성 가이드라인**: 중첩 수량자 금지, 모든 수량자에 구체적 문자 클래스 사용 의무화
>
> **확장 생각**: 실제로 Cloudflare, AWS WAF 등의 상용 WAF는 이러한 트레이드오프를 고려한 맞춤형 엔진을 사용합니다. Google의 RE2 라이브러리는 DFA 기반으로 성능을 보장하면서도, 실용적인 수준의 기능을 제공하여 많은 시스템에서 채택되고 있습니다.
> </details>
